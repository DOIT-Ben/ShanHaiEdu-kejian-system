# 工作流与创作引擎

## 1. 两类引擎、一个任务底座

系统包含两种编排，但共用生成任务、资产、审核和事件底座：

- **项目工作流引擎**：按课时和分支推进教材、教案、PPT、视频和交付。
- **通用创作引擎**：按创作条目批量生成图片、视频或 PPT，不要求绑定项目。

项目节点通过不可变 `CreationPackage` 把输入交给通用创作引擎；创作结果通过 `SaveToProjectOperation` 显式返回项目。两者不能直接共享可变数据库对象。

## 2. 工作流定义与运行

`WorkflowDefinitionVersion` 是发布后不可变的有向无环图，节点定义包含：

- 稳定 `node_key`、名称、用户语言和分支；
- 输入合同、输出合同和内容定义引用；
- 前置条件、批准条件、质量门槛；
- 自动推进能力和所需权限；
- 重试、超时、取消和补偿策略；
- 生成创作包、保存资产和失效传播规则。

创建项目时固定工作流版本和内容发布版本。管理员发布新版只影响新项目；现有项目迁移必须先做差异预检，再由有权限者确认。

`WorkflowRun` 对应项目的一次主运行；每课时每分支形成 `BranchRun`；具体执行实例为 `NodeRun`。节点可重跑，但历史运行不可覆盖。

## 3. 统一节点状态

节点状态只允许以下值：

| 状态 | 含义 |
| --- | --- |
| `disabled` | 分支关闭或配置不适用 |
| `not_ready` | 前置条件未满足 |
| `ready` | 可开始，等待用户或自动策略 |
| `draft` | 已有可编辑内容，尚未提交 |
| `queued` | 异步任务已入队 |
| `running` | 正在执行 |
| `review_required` | 有可审核结果 |
| `approved` | 当前版本已批准 |
| `partially_completed` | 批量条目部分完成，可继续处理 |
| `failed` | 本次执行失败且未自动恢复 |
| `paused` | 自动化主动暂停，等待选择或风险确认 |
| `cancel_requested` | 已请求取消，等待 Worker 确认 |
| `cancelled` | 本次运行已取消 |
| `stale` | 基于已被替代的上游批准版本 |
| `skipped` | 用户明确跳过可选节点并留下原因 |

节点状态不是生成任务状态、审核状态或保存状态。前端可聚合展示，但数据库分别记录，避免一个状态字段承担所有语义。

## 4. 一个工作流内核与两种执行方式

### 按步骤辅助创作 `guided`

系统准备输入、提示词和校验，在主要节点形成结果后暂停；用户可以编辑、采用、审核和推进。教师需要重点参与哪些节点由策略声明，不复制工作流。

### 一键生成 `automatic`

系统可自动创建草稿、编译提示词、执行节点、按策略采用候选、写回项目和批准低风险中间产物，但必须满足：

- 结构化输出通过 Schema 和业务规则；
- 质量分达到内容包规定的阈值；
- 实际成本和预计后续成本未超预算；
- 未命中人工必审节点；
- 供应商结果没有安全或版权风险标记；
- 自动选择证据、策略版本和理由被持久化。

最终教案、最终 PPTX、最终课堂视频默认仍由教师确认。组织管理员可在策略允许范围内调整必审点。

两种方式引用同一个`WorkflowDefinitionVersion`、`WorkflowRun`、`NodeRun`、任务、版本、依赖、质量门禁和资产。所谓“只自动一部分”由节点级策略覆盖表达，不形成第三种模式或另一套数据。

迁移旧值时，`manual`映射为`guided`并关闭所有节点自动动作，`assisted`映射为`guided`且把原行为保存在节点规则中，`automatic`保持不变。迁移只影响后续策略解析；已经固化在`NodeRun`和`WorkflowRun`中的历史策略快照不可改写。

## 5. AutomationPolicy

自动化策略不是若干前端开关，而是受版本管理的规则集合。至少包含：

- `mode`：`guided / automatic`；
- 各节点 `auto_start`、`auto_submit`、`auto_approve`；
- 候选数量、选择算法和最低质量分；
- 最大尝试次数、单节点成本和项目预算；
- 人工必审节点与暂停条件；
- 错误策略：重试、切换路由、暂停或跳过；
- 允许自动保存回项目的槽位类型。

每个 `NodeRun` 保存实际使用的策略快照，不受日后管理员修改影响。

## 6. 项目主流程状态推进

### 教材与课时

1. 确认上传后解析教材，保存页、段落、图片和引用坐标。
2. 课时划分节点只输出课时数、顺序、标题、范围、目标摘要和理由，不混入完整教案。
3. 教师调整并批准课时划分，系统创建或同步 `LessonUnit`。
4. 每课时创建教案和三类九套导入设计两个兄弟产物，并按选择开启 PPT、视频分支。

### 教案

1. 读取批准的课时划分、教材引用和当前教案内容定义。
2. 编译可见、可编辑的最终提示词。
3. 生成结构化教案草稿，默认十二部分，但以项目固定的内容定义为准。
4. 前端按字段配置展示、校验和编辑；允许删除的字段可删，必填字段给出原因和替代方案。
5. 教师批准后创建当前批准版本；PPT 默认读取该批准版本，视频分支不读取教案正文。

### 三类九套导入设计

1. `intro.generate_options` 读取批准课时、知识点、学习目标、内容边界、不得提前讲授、教材证据摘要和可选教师偏好。
2. 节点一次生成最终一套或三类九套方案；科普、应用、故事是主要倾向，辅助倾向可以交叉。
3. 方案集形成独立 `intro_option_set` 产物，可在教案导出中作为正文附录；教案与方案集分别批准。
4. 教师选择形成 `intro_selection` 运行记录，不覆盖方案集；自动化只能在策略允许时采用唯一最高分。
5. 视频获得被选方案完整快照，不获得教案正文、教学目标、教学过程、板书或作业。

### PPT

严格执行：大纲 → 页级设计稿 → 封面候选 → 视觉风格合同 → 正文资产 → 页面预览与编辑 → 可编辑 PPTX → 审核。

封面必须单独审核。正文固定纯白背景，但不等于无设计：每页都要有图片主视觉，标题、正文、图形、公式、版式和留白由页级四层设计控制。每页是独立产物，可重做单页而不重跑全套；详细字段和质量门见 [PPT 生产流程与页级合同](../workflows/PPT_PRODUCTION.md)。

### 视频

严格执行：选定导入方案 → 母版剧本 → 粗分镜 → 视觉母图/风格合同 → 图片资产 → 细分镜及视频提示词 → 候选片段 → 每镜头选定合格片段 → 配音/音效 → FFmpeg 合成 → 成片审核。

视频分支不等待教案批准。它严格继承所选方案中的课程驱动创意、钩子、课程关联、课堂首问、交接时刻和不得提前讲授内容，不再读取或重新解释教案与教材。

母版剧本包含完整故事、场次、动作、旁白、对白与声音意图，并根据故事结构推荐60至180秒总时长；教师确认服务端费用估算后才批准母版。粗分镜把它拆为节拍并记录预计时长和资产需求。视觉母图锁定主风格后，人物、场景、道具和生物按独立资产逐项生图。只有图片资产通过后，细分镜才为每个shot编译6至30秒模型提示词和精确垫图引用。完整字段与验收见[课堂导入视频生产流程](../workflows/VIDEO_PRODUCTION.md)。

## 7. 通用创作包

项目向创作中心导出时，`CreationPackage`必须达到`ready`，并包含：

- 包类型：image / video / ppt；
- 来源项目、工作流运行、课时、节点运行和版本；
- 一项或多项 `CreationPackageItem`；
- 每项的结构化提示词、负面约束、参考资产和输出规格；
- 风格合同和跨条目一致性键；
- 推荐候选数量和模型能力要求；
- 固定的来源项目目标槽位及覆盖策略；
- 内容、提示词、上下文和策略快照 ID。

创建包后若项目上游变化，旧包仍可复现，但标记 `stale_source`；用户须确认继续或重新导出。

## 8. 通用创作批次

创作中心把包实例化为`source_kind=project`的`CreationBatch`；该批次必须保留`creation_package_id`及来源字段，不能改换项目或槽位。直接从创作台建立的批次使用`source_kind=standalone`，不携带伪造的项目上下文。

用户可以：

- 修改任一条目业务提示词、参考图、逻辑质量/速度档位和输出规格；真实Provider由服务端模型网关路由；
- 新增、复制、删除或重新排序条目；
- 单项生成、选择若干项并发或一键生成全批次；
- 为单项或整个批次停止、重试、切换逻辑质量/速度档位；
- 比较并采用候选；项目批次可把已采用结果写回固定槽位，独立批次可下载、保存到个人资料或显式选择有权限项目写回。

修改条目产生 `CreationItemRevision`。每个生成任务固定引用某次修订，运行中编辑只影响下一次生成。

## 9. 四个创作命令

- `save_prompt_version`：保存业务提示词、参考资产和输出规格修订，不触发生成。
- `generate`：固定提示词版本、上下文、参考资产和逻辑模型档位，创建`PromptSnapshot`与`GenerationJob`。
- `adopt`：追加`Adoption`记录，选择本批次后续使用的候选，不写入项目。
- `save_to_project`：把已采用候选幂等写入项目槽位。项目批次目标来自`CreationPackage`；独立批次必须显式提交目标并重新鉴权。

四个命令分别鉴权、审计和返回稳定错误，不允许一个命令隐式触发另一个命令。

现有公开端点或字段若把采用、写回和提示词修订混在一起，必须先增加新合同和兼容适配，再标记旧入口弃用；不得在同一版本直接删除或静默改变旧请求的含义。

## 10. 生成任务状态机

`GenerationJob` 状态：`created` → `queued` → `running` → `succeeded | partially_succeeded | failed | cancelled`。

每次供应商调用是 `GenerationAttempt`。持久化状态为 `running`、`succeeded`、`failed`、`cancelled`、`submission_unknown`；视频供应商的 `submitted`、`polling` 等操作状态仍保存在供应商中立结果中。重试新建 Attempt，不覆盖旧响应。

每个候选由`GenerationResult`承载：`available`、`rejected`、`expired`。采用由独立`Adoption`记录，项目归属由`SaveToProjectOperation`和`AssetBinding`记录，不把三种状态压进候选字段。

## 11. 并发、租约和回调

- 创建任务必须带 `Idempotency-Key`；相同用户、作用域、请求摘要返回同一任务。
- Worker 领取 Attempt 写 `lease_owner`、`heartbeat_at` 和 `lease_expires_at`，定期续租；过期 Attempt 由数据库行锁竞争恢复，失去 owner 的 Worker 不得续租或写终态。
- `video_submit` 或历史操作类型无法判定的过期 Attempt 进入 `submission_unknown`，不得自动重提；其他安全操作可终结为租约过期失败后由上层显式重试。
- Attempt 终态与其唯一 `UsageRecord` 在同一事务写入；已有用量记录的历史 Attempt 恢复时不得重复插入或重复计费。
- 同一创作条目允许并发候选任务，但同一节点的自动推进使用分布式锁和数据库唯一约束双保险。
- 供应商回调按 `provider + provider_task_id + event_id` 去重，乱序回调按状态优先级处理。
- 取消采用协作式取消；供应商不能取消时，结果可到达但标记为取消后结果，不自动保存。
- 重试采用指数退避和随机抖动；参数错误、内容拒绝和余额不足不盲目重试。

## 12. 精确失效与重跑

上游新版本批准时：

1. 读取旧当前批准版本到下游的 `ArtifactRelation`。
2. 由拥有上游业务语义的analyzer生成经验证的`StaleImpactSelection`，再与关系的`impact_scope`精确匹配真实受影响边。
3. 标记对应产物和节点 `stale`，写影响摘要。
4. 不自动删除用户修改；提供“只重生成受影响部分”“全部重做”“确认沿用”三种动作。
5. `automatic`仅在内容定义明确允许且没有人工编辑冲突时执行局部重跑。

## 13. 失败恢复

- 文本结构校验失败：先执行有限次数修复提示，再进入人工编辑。
- 单页 PPT 或单镜头视频失败：分支为 `partially_completed`，其余成功项可继续使用。
- 供应商不可用：模型网关按能力和政策切换路由，不更改业务输入快照。
- 应用重启：调度器从数据库扫描未完成任务和过期租约恢复。
- 保存回项目失败：创作结果保留，保存操作可幂等重试。
- SSE 丢失：前端按事件游标补取，并以 REST 查询结果为最终状态。
