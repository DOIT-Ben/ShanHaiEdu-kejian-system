# 内容运行时与模型网关

## 1. “血肉”如何进入系统

提示词、教案结构、工作流、三类九套导入锚点、PPT 手册和质量规则不是散落在代码中的字符串，而是由管理端导入、测试、发布并由项目固定版本的 **内容包**。代码提供稳定运行时，内容包提供可演进的业务能力。

内容包只允许声明式 JSON、YAML、Markdown、模板和测试样例，不允许上传并执行任意代码。

## 2. 内容类型

| 类型 | 用途 | 示例 |
| --- | --- | --- |
| `input_definition` | 教师输入、系统补全、上下文注入和UI可见性 | 图片需求、PPT页数与风格选择 |
| `content_definition` | 输出字段、层级、必填、编辑和导出规则 | 教案十二部分及其子结构 |
| `prompt_template` | 可组合提示词模板 | 课时划分、教案、PPT 大纲 |
| `rule_set` | 业务校验和质量门槛 | 教案必填、视频时长限制 |
| `workflow_definition` | 节点、依赖、分支和自动化能力 | 小学数学主流程 |
| `anchor_library` | 三类九套导入设计及锚点定义 | 科普、应用、故事 |
| `style_preset` | 视觉风格、版式和一致性约束 | PPT 封面、视频动画风格 |
| `model_profile` | 逻辑能力和默认参数 | 中文教案文本、16:9 生图 |
| `handbook` | 给模型和管理者使用的规范内容 | PPT 工作流手册 |
| `evaluation_suite` | 样例输入和预期检查 | 百分数教材回归样例 |
| `projection_template` | 结构化产物到教师可读文件的确定性投影 | 教案正文、视频节点文档 |
| `generation_template` | 组合输入、风格、Prompt、输出、投影和逻辑模型能力 | 图片提示词、PPT页级设计稿 |

## 3. 标准导入包

```text
package.zip
├── manifest.json
├── capabilities/
├── inputs/
├── generations/
├── definitions/
├── schemas/
├── prompts/
├── rules/
├── workflows/
├── anchors/
├── styles/
├── model-profiles/
├── handbooks/
├── evaluations/
├── projections/
└── assets/
```

`manifest.json` 必须声明格式版本、包 ID、语义版本、兼容的运行时版本、内容项类型、Schema ID、相对路径、语义内容哈希、依赖包、入口项和变更摘要。内容项只允许 POSIX 相对路径，禁止绝对路径、反斜杠和 `..`。JSON 内容哈希基于 UTF-8 的确定性键排序序列化，不受缩进和换行影响；原始 ZIP 哈希作为独立存储完整性证据。

导入流程为：上传 → 解压隔离 → 路径与哈希检查 → Schema 校验 → 引用解析 → Context 来源白名单检查 → 静态安全检查 → 样例测试 → 生成预览 → 管理员发布。

包版本发布后不可变。多个包通过 `ContentRelease` 组合为一次可用发布；项目创建时固定 `content_release_id`。禁止“修改当前模板后所有老项目静默变化”。

当前正式发布事实为`shanhai.primary_math.courseware@1.0.0`。#130与PR #141已经把`1.1.0`前向候选合入主线，#146继续修复其安全与发布固定性缺口；#146合并后，通过`python -m apps.api.cli publish-golden-content`执行PostgreSQL前向发布，发布及验证通过后才视为正式发布并成为新项目默认。服务先复用上述路径、Schema、哈希、引用和上下文白名单校验，再在一个数据库事务中写入Manifest、全部内容项、内容定义投影、工作流节点目录、Release和追加式运行时默认版本。项目创建从数据库读取最新默认版本；重复执行同一包只校验并返回既有发布，内容或哈希冲突则拒绝，既有`1.0.0`项目不会被重新绑定。

旧v1绑定可以继续承载原有工作流；只有输出投影消费者请求其未声明的v2 `output_persistence`时，才稳定返回`WORKFLOW_RELEASE_UNSUPPORTED`。缺失投影不能被解释为业务默认值，也不能被扩大为旧工作流整体无法加载或启动。

内容定义发布时把字段树编译为可执行Draft 2020-12 JSON Schema，同时保存标准Schema无法表达的跨字段规则；Artifact提交校验会合并两类错误。未知规则必须阻止发布，不能作为空规则静默进入运行时。

### 现有 ShanHai Skills 的导入映射

管理端提供 `ShanHai Skill Bundle` 导入适配器，把现有技能包规范化为内容包，不要求运营人员手工拆文件：

| Skill 内容 | 运行时映射 |
| --- | --- |
| `SKILL.md` | `generation_template` 的职责、模式、方法和硬门禁 |
| `agents/*.yaml` | 展示名、触发语和默认入口元数据 |
| `assets/*.schema.json` | 正式输入/输出合同与动态内容定义 |
| `references/*.md` | 分阶段 handbook、rule_set 和提示词材料 |
| 注册表与上下游 | 能力图、输入输出类型和工作流引用 |
| 测试样例 | `evaluation_suite` |
| 确定性渲染规则 | `projection_template` 或已注册 renderer ID |

上传包中的 Python、Shell 或可执行文件只做隔离扫描和人工审计，不在业务运行时直接执行。若某个校验器或渲染器确实需要代码，必须作为经过评审、签名和部署的运行时扩展发布，内容包只引用已注册扩展 ID。这样既保留 Skills 的业务“血肉”，又不把管理端变成远程代码执行入口。

导入预览必须展示：将新增或替换哪些能力、Schema、提示词、规则、工作流、锚点和评测；哪些文件被忽略或隔离；与当前 Release 的破坏性差异。管理员确认后才发布。

### Schema 驱动生成模板

每个 V1 内容项使用共同信封 `api_version + kind + metadata + spec`。Manifest 负责稳定键、类型、路径、Schema 和哈希；对应类型的 JSON Schema 只校验 `spec`。跨条目引用使用结构化的 `item_key + kind`，不在导出包中暴露数据库 UUID。

一个 `GenerationTemplate` 至少固定引用：

- 一个 `InputDefinition`；
- 一个 `PromptTemplate`；
- 一个 `ContentDefinition` 输出合同；
- 零个或多个 `StylePreset`；
- 一个或多个 `ProjectionTemplate`，并标明教师视图、Provider请求或导出用途；
- 一个逻辑模型能力，不引用具体供应商模型。

后台字段通过输入定义和内容定义描述；前端只消费其中的标签、可见性和控件提示。主要字段可直接展示，次要或高级字段按需展开，系统与上下文字段保持隐藏。教师始终可以查看并修改一段完整业务提示词或可读作品，但不会收到内部Prompt组件、输出Schema、校验修复规则或Provider格式。

图片提示词可以先生成通过 Schema 的内容、构图、色系、质感、光线、风格和负面约束对象，再由 `ProjectionTemplate` 渲染为一段完整提示词。PPT设计稿可以先保存页面类型、教学目标、视觉风格、背景策略、版式、可编辑元素和资产需求，再投影为Markdown、预览和PPTX。

### 工作流节点生成绑定

#130与PR #141已经把版本化`WorkflowNodeGenerationBinding v2`候选合入主线，用于连接工作流图、生成模板和运行快照；#146与PR #148已经完成不可变快照、语义校验和发布边界加固。工作流图只负责顺序；`GenerationTemplate`只负责如何生成；节点绑定负责本节点能怎样执行、能读取什么、怎样校验和何时审核；运行快照记录本次真正使用的版本和值。候选目录显式记录47节点的scope/branch/入口/依赖、validator descriptor、Artifact/CreationPackage输出投影和质量报告/人工门禁声明；既有项目和已发布`1.0.0`绑定仍保持不变。

每个节点必须显式声明：

- `execution_kind`：`model_generation`、`deterministic`或`human_gate`；
- 模型节点的逻辑能力与`GenerationTemplate`，或确定性节点的平台安全执行器；
- 输入/输出合同、服务端规范引用、Context策略和参考资产策略；
- 教师业务Prompt是可编辑还是隐藏；内部模板、输出合同、校验修复和Provider格式始终`server_only`；
- validator、有限修复模板和下游批准策略。
- 模型节点的`output_persistence`。编译器只允许受限JSON Pointer和固定常量，先形成Artifact；只有声明`creation_package`的模型节点才在Artifact版本落库后形成CreationPackage，确定性与人工门禁节点不走该路径；质量报告只在声明的producer→report链成立时进入预期索引。

Binding v2按ContentDefinition形成的反向索引只表达固定Release/Workflow下的预期producer，以及声明存在时对应的validator report和gate；它不证明某个NodeRun、质量报告或人工批准已经实际执行。

### 通用模型节点执行闭环

通用执行器只消费项目已固定的ContentRelease、WorkflowDefinitionVersion、GenerationTemplate和WorkflowNodeGenerationBinding，不按教案、PPT、图片、视频或TTS类型分支。T1在短事务内冻结PromptSnapshot、ContextSnapshot、参考资产、上游ArtifactVersion、外部幂等请求和NodeRun owner；Provider调用在事务外执行。确定性Fake与真实Adapter共享ModelGateway，但普通CI只调用Fake。

文本Gateway先返回内存中的pending结果。发布Output Schema校验成功后，Attempt终态、唯一UsageRecord和`node_execution_recovery_facts`在同一事务写入；该私有事实只保存已校验结构化JSON、规范哈希、Schema摘要、快照ID/哈希、租户/项目/节点/Attempt/request/owner绑定、1 MB上限和24小时过期时间，不进入OpenAPI、Attempt/Usage正文或日志。checkpoint回滚时三者全部回滚；checkpoint成功后Worker崩溃，下一Worker只消费恢复事实，不重复调用Provider或计费。

最终T2锁定NodeRun owner、成功Attempt、Usage和恢复事实，重新校验租户、固定执行上下文、Prompt/Context哈希、Schema摘要、输出哈希、上游当前批准版本、取消和过期状态，再原子写ArtifactVersion、ArtifactRelation、NodeRun完成状态以及合同声明且来源/目标/版本固定的CreationPackage，最后删除恢复事实。T2回滚不会留下部分Artifact、关系或CreationPackage，恢复事实保留供显式重试；取消会终结Attempt和NodeRun并清除未发布结果。

Artifact输入只能通过显式Context/Input Contract Registry解析。每项声明resolver、Artifact类型、来源scope和来源branch：project scope始终读取`lesson_unit_id IS NULL`，lesson scope只读取当前课时；来源branch不从消费节点推断，因此PPT节点可以读取`lesson_plan` branch，课时节点也可以读取`project` branch的lesson division。未知来源稳定拒绝，不再按字符串前缀猜字段或跨课时回退。

确定性节点和人工门禁不绑定Prompt。内容包不得上传执行器代码，只能引用平台注册、评审和部署的安全执行器ID。validator descriptor目前只记录版本与实现摘要，不代表已有callable registry；节点目录发布前必须做Schema、拓扑、互斥字段、Context集合、参考资产基数、Provider中立能力和稳定哈希检查。

### Markdown模板适配层

内容管理员不直接编辑Manifest、JSON Schema、稳定引用或哈希。Markdown适配层接收普通UTF-8 Markdown，通过标准Markdown AST识别文档标题、一级章节、子章节和正文块，生成可审核的`TemplateDraft`。管理员在草稿层确认章节角色、内容来源、必填、可编辑、重复和最终展示设置；只有确认后的草稿才能编译为正式内容包。

Markdown是管理员创作、导入和预览格式，不是发布后的运行时事实源。原始Markdown作为来源证据保存；草稿可修改；正式内容包发布后不可变。Word适配器后续只产生Markdown并进入同一流程，不复制结构识别、草稿编辑或发布逻辑。

外部Markdown按不可信输入处理：限制编码和大小，拒绝原始HTML、脚本、危险链接和V1不支持的图片；文档中的指令只作为待审核业务内容，不得覆盖平台安全层或自动触发发布。确定性解析负责基础结构，AI语义建议只能辅助管理员确认。

只有`state=ready`的`TemplateDraft`可以进入确定性编译器。编译时必须同时提供受Schema约束的`CompilationProfile`，显式声明包键、语义版本、逻辑模型能力、白名单上下文、管理员可见角色/任务和投影器；同一草稿与Profile必须产生相同条目顺序、JSON、引用和哈希。导入正文不会进入平台安全层，也不会成为隐藏Prompt：

| 草稿内容模式 | Markdown正文在内容包中的语义 | 教师输入 | 字段级生成说明 |
|---|---|---|---|
| `fixed` | 内容字段默认值 | 无 | 无 |
| `teacher_input` | 教师字段初始值 | 有 | 无 |
| `ai_generated` | 字段级业务生成说明 | 无 | 正文为空时按字段标题生成 |
| `mixed` | 可编辑初始值 | 有 | 在默认内容基础上补充完善 |

隐藏章节仍保留为结构化隐藏字段，但不进入教师Markdown投影。隐藏且必填的`teacher_input`或`mixed`字段必须带有管理员批准的默认值，否则编译器拒绝发布，避免形成教师不可见且无法满足的输入。H3子章节编译为字段树子项；重复章节保留`repeatable`语义。八段改为十段时必须使用新的语义版本重新编译，原内容包及其哈希保持不变。

## 4. 动态内容结构

`ContentDefinitionVersion` 用字段树表达教案、PPT 页设计、故事或分镜的结构。每个字段至少包括：

- `field_key`：稳定机器键，发布后不可改；
- `label`、`description`、`type`；
- `required`、`editable`、`deletable`、`repeatable`；
- `min_items`、`max_items`、长度或数值限制；
- `ui_widget`、占位说明、排序和折叠规则；
- `generation_instruction`、`validation_rules`；
- `export_mapping`：映射到 DOCX、PPTX、PDF 或视频脚本；
- `visibility_condition`：基于项目分支或其他字段的声明式条件。

默认教案结构为：教学内容、教材分析、学情分析、设计意图、教学目标、教学重难点及突破策略、教学准备、教学过程、板书设计、课堂总结、分层作业、教学反思。它是首发内容包默认值，不是代码常量。管理员可导入新版字段和子结构；旧项目继续使用固定版本，也可显式迁移。

教师可以修改一次生成的业务要求和产物字段值，但不能通过提示词修订删除必填字段、改变字段类型或把十二段改为十段。结构调整属于内容管理员发布新的`ContentDefinition`、`PromptTemplate`和`GenerationTemplate`版本；旧项目继续固定旧版本，除非显式迁移。

三类九套拥有独立的 `intro_option_set` 内容定义，不作为教案字段树的固定子字段。教案导出器可以把已批准方案集投影为正文后的附录，但数据库、审核和版本始终分离。

### 单一语义源与双输出

每个正式内容产物先保存通过 Schema 的结构化 JSON，教师编辑器直接修改同一结构。Markdown、DOCX、PDF、PPTX 预览、视频节点文档等由受版本管理的确定性投影生成，不再让模型自由写第二份“教师版”。

投影文件记录 `source_artifact_version_id`、`projection_version_id` 和内容哈希。源 JSON 修改后，旧投影标记过期；只有重新投影且哈希检查通过才可交付。内部 ID 和枚举可转成人类标签，但正文不能被投影器摘要或改写。

## 5. 上下文绑定与隔离

每个提示词模板声明可读取的 `ContextBinding`，运行时按白名单装配上下文。允许的来源示例：

- `material.approved_parse`
- `lesson_division.approved_version`
- `lesson_plan.approved_version`
- `project.teacher_preferences`
- `asset_slot.lesson.02.video.style_master`

运行时同时校验来源已注册、调用方只提交模板声明来源、必填绑定、条目数和字节上限；任一检查失败都必须在调用模型前停止。ContextSnapshot按稳定来源版本排序并计算语义哈希，Provider不能自行补读数据库或未声明资产。

服务端严格区分三种约束输入：

1. `instruction_policy`引用随内容发布固定的handbook、rule_set、anchor_library和style_preset；这些内容不作为教师Prompt组件公开。
2. `context_policy`限制PromptTemplate的ContextBinding允许集合与明确禁区；具体必填、展示级别和大小限制仍由ContextBinding声明，两层必须同时通过。
3. `reference_asset_policy`只管理实际传给多模态模型的文件版本，采用`none / optional / required`并限制角色、媒体类型、基数、顺序、来源和Provider暴露方式。

Context不能偷渡参考资产，参考资产也不能让Provider自行读取项目数据库。三类输入分别形成版本引用、内容哈希和运行快照。

课程驱动导入节点只读取最小课程种子和教师偏好：批准课时、知识点、学习目标、内容边界、不得提前讲授、年级/年龄和教材证据摘要；禁止读取完整 `lesson_plan`、PPT和视频产物。节点一次输出最终方案集。视频节点只读取被选方案快照，禁止读取完整教材、教案和PPT。

## 6. 提示词编译

最终请求提示词按固定层次编译：

1. 平台不可修改的安全和租户规则；
2. 当前节点角色与任务边界；
3. 内容包中的业务方法和手册；
4. 动态内容结构与输出合同；
5. 白名单允许的上游批准版本和项目资产；
6. 教师选择、偏好和本轮返修意见；
7. 机器可校验的 JSON Schema；
8. 供应商适配层的格式要求。

服务端从第2至第6层中只选择模板明确标记为教师可见、可编辑的内容，投影为一段完整“业务提示词”。教师端不接收第1、7、8层，也不接收隐藏的内部方法、handbook、Context来源/哈希、输出Schema或校验修复规则。提交生成时仍创建完整`PromptSnapshot`，记录所有层、模板版本、变量摘要、用户修改差异、上下文快照、输出Schema和最终请求哈希；公共投影与内部快照是两个不同的权限面。

教师公共Prompt Preview只返回快照身份与哈希、完整业务提示词，以及PromptTemplate确定的编辑模式和字符上限。服务端读取旧`preview_json`时按该白名单重新投影；历史记录中的Schema、锁定层和Context摘要不会进入公共响应。

教师修订只替换允许编辑的业务层。平台安全、结构合同、必填字段、校验修复和Provider适配始终高于本次自由文本，不能被删除或覆盖。

Prompt领域的`user_edit_policy.mode`唯一取值是`replace_editable_layer`：教师修订作为一次不可变替换差异写入，并只替换模板标记为可编辑的业务层。Prompt模板、Markdown编译配置和运行时不得声明或解释追加式修订；历史内容包若出现未知模式，必须在发布或编译边界拒绝。CreationPackage与项目资产绑定中的`append`是独立的追加历史语义，不属于Prompt修订合同，继续按各自合同执行。

结构化 Prompt 组件和教师最终看到的完整提示词同时保留。教师修改完整提示词时创建不可变 `PromptRevision`，本次调用以该修订为最高优先级；系统保存基础结构、基础投影哈希和修订差异，不把自由文本反向解析回字段。基础输入、模板或投影变化后，旧修订标记为基于旧基础，必须明确沿用或重新合并。

返修默认形成新提示词修订并重新生成，不对旧模型响应做不可追溯的覆盖。对支持多轮编辑的供应商，也必须把此前结果和新指令固化到新快照。

## 7. 结构化输出与质量门槛

模型输出依次经过：

1. 供应商响应适配；
2. JSON 提取和 Schema 校验；
3. 受限的格式修复；
4. 业务规则校验；
5. 安全和敏感信息检查；
6. 内容质量评分；
7. 保存规范化结果、校验报告和必要的不可逆响应摘要；当前不保存原始响应。

修复不能偷偷改变业务语义。超过修复次数后，节点进入可编辑草稿或失败状态，不得把不合格内容伪装成成功。

## 8. 模型网关定位

模型网关把业务所需能力与具体供应商解耦。业务模块只请求逻辑能力，例如：

- `text.structured.zh_primary_math`
- `image.generate.education_16x9`
- `video.image_to_video.6s_30s`
- `audio.tts.child_friendly_zh`

网关负责选择供应商、参数转换、限流、重试、熔断、回调、计量和统一错误；它不决定教案结构、故事方法或审核规则。

当前基础层为文本、图片和视频分别提供判别明确且拒绝额外字段的请求、结果与Provider Port。媒体引用只使用不可变文件版本，生成结果只返回对象存储文件事实，不允许把Provider URL、私有参数或原始响应带入平台合同。普通CI使用确定性Fake；真实图片和视频Adapter、供应商私有状态映射及受控冒烟由后续独立任务实现。

视频调用使用`submitted / polling / succeeded / failed / cancelled / submission_unknown`状态。每次提交、轮询和取消都是独立Attempt；提交成功时在返回调用方前持久化`provider_task_id`，后续轮询复用该任务标识。无法确认供应商是否已收到提交时进入不可自动重提的`submission_unknown`，避免重复生成和扣费。运行中Attempt使用owner、心跳和过期时间租约；恢复Worker通过PostgreSQL行锁竞争终结过期记录，未知提交保持不可重提，失去租约的旧Worker不得续租或覆盖终态。

### 阿里云主要开发环境入口

ShanHaiEdu开发任务只连接统一公网入口`https://newapi.doitbenai.cloud/v1`，不直接连接NewAPI、LiteLLM或`media_router`。统一业务密钥覆盖当前公开模型：文本`deepseek`、`gpt-5.6`，图片`image-2`、`nanobanana`，视频`video-grok`。图片生成和视频提交必须提供稳定且唯一的`Idempotency-Key`；视频使用`POST /videos`提交、`GET /videos/{taskId}`轮询和`GET /videos/{taskId}/content`下载。

阿里云开发用户从仓库外受控文件加载密钥和非敏感路由，不在仓库、worktree、镜像或Compose文件中复制：

```bash
set -a
. /srv/shanhaiedu/runtime/secrets/model-gateway.env
set +a
```

服务器接入说明位于`/srv/shanhaiedu/runtime/docs/SHANHAIEDU-MODEL-GATEWAY.md`，机器OpenAPI快照位于`/srv/shanhaiedu/runtime/docs/model-gateway-openapi.json`。运行真实任务前仍应读取公网`/healthz`和`/openapi.json`核验新鲜度；快照不是版本永久固定值。

当前代码只实现OpenAI兼容真实文本适配器，因此`SHANHAI_TEXT_PROVIDER_*`映射到统一入口，并通过`SHANHAI_TEXT_PROVIDER_SECRET_ENV=MODEL_GATEWAY_API_KEY`间接读取密钥。图片和视频Provider Port已经存在，但真实适配器必须由各自Issue实现和冒烟，不能仅凭模型目录可见就宣称媒体出口完成。普通CI和常规单元、合同、集成测试不得加载该文件。

## 9. 模型配置模型

- `ModelProfileVersion`：逻辑能力、输入输出模态、参数 Schema、成本单位、质量标签和数据政策。
- `ProviderConfig`：供应商、区域、端点、密钥引用、并发限制、回调能力和启用状态。
- `ModelRoute`：逻辑能力到供应商模型的主路由、备路由、条件和权重。
- `RoutingPolicy`：按组织、节点、质量、预算、地区、文件大小和降级策略路由。

密钥只保存为 Secret Manager 的 `secret_ref`，不进入数据库明文字段、日志、导出包或前端。管理端可以测试连接、查看脱敏配置和逐步启停路由。

## 10. 路由、重试与熔断

路由顺序：能力匹配 → 租户允许 → 数据政策 → 输入规格 → 预算 → 健康度 → 质量优先级 → 权重。系统必须记录最终为什么选择该路由。

- 网络超时、5xx、限流可按策略重试或切备路由。
- 参数错误、内容安全拒绝、无余额不在同一路由盲重试。
- 熔断器按供应商模型和错误类型统计，半开探测不占用正常用户任务。
- 切换供应商时保持业务输入快照不变，只产生新的 Attempt 和供应商请求格式。

## 11. 用量、预算与审计

每次 Attempt 写 `UsageRecord`：组织、用户、项目、节点、逻辑模型、供应商、输入输出单位、计价版本、估算/实付成本、币种和延迟。完整Prompt、模型正文、密钥、Provider私有字段和原始响应不进入Attempt、Usage、日志或公共合同；以后若确需诊断留存，必须另行决定隔离存储、权限和删除期限。

Fake与真实Provider都通过ModelGateway的同一审计端口。端口只接收请求ID、能力、最终请求哈希、路由摘要、用量和稳定错误码，不接收完整提示词、模型正文、密钥或Provider原始错误；失败Attempt仍写零或已知用量记录，避免成本与失败率出现审计空洞。

预算检查分三层：组织月度限额、项目预算、单节点/单批次限额。超出预计成本时`guided`和`automatic`都必须暂停并让用户确认，自动化策略不能绕过预算门禁。

## 12. 管理端发布治理

内容中心和模型中心至少支持：

- 导入、校验、预览、测试、发布、停用、回滚到旧发布；
- 版本依赖和受影响项目预览；
- 只读查看线上使用的提示词与结构快照；
- 灰度到指定组织或测试项目；
- 双人复核高风险模型路由和平台安全规则；
- 完整审计：谁在何时导入、测试、发布、启停或迁移。

“回滚”是把新项目默认发布指针切回旧的不可变版本，不是修改或删除已发布内容。
