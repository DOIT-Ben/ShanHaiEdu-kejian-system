# 内容运行时与模型网关

## 1. “血肉”如何进入系统

提示词、教案结构、工作流、三类九套导入锚点、PPT 手册和质量规则不是散落在代码中的字符串，而是由管理端导入、测试、发布并由项目固定版本的 **内容包**。代码提供稳定运行时，内容包提供可演进的业务能力。

内容包只允许声明式 JSON、YAML、Markdown、模板和测试样例，不允许上传并执行任意代码。

## 2. 内容类型

| 类型 | 用途 | 示例 |
| --- | --- | --- |
| `input_definition` | 教师输入、系统补全、上下文注入和UI可见性 | 图片需求、PPT页数与风格选择 |
| `content_definition` | 输出字段、层级、必填、编辑和导出规则 | 教案十二部分及其子结构 |
| `prompt_template` | 可组合提示词模板 | 课时划分、教案、PPT 大纲 |
| `rule_set` | 业务校验和质量门槛 | 教案必填、视频时长限制 |
| `workflow_definition` | 节点、依赖、分支和自动化能力 | 小学数学主流程 |
| `anchor_library` | 三类九套导入设计及锚点定义 | 科普、应用、故事 |
| `style_preset` | 视觉风格、版式和一致性约束 | PPT 封面、视频动画风格 |
| `model_profile` | 逻辑能力和默认参数 | 中文教案文本、16:9 生图 |
| `handbook` | 给模型和管理者使用的规范内容 | PPT 工作流手册 |
| `evaluation_suite` | 样例输入和预期检查 | 百分数教材回归样例 |
| `projection_template` | 结构化产物到教师可读文件的确定性投影 | 教案正文、视频节点文档 |
| `generation_template` | 组合输入、风格、Prompt、输出、投影和逻辑模型能力 | 图片提示词、PPT页级设计稿 |

## 3. 标准导入包

```text
package.zip
├── manifest.json
├── capabilities/
├── inputs/
├── generations/
├── definitions/
├── schemas/
├── prompts/
├── rules/
├── workflows/
├── anchors/
├── styles/
├── model-profiles/
├── handbooks/
├── evaluations/
├── projections/
└── assets/
```

`manifest.json` 必须声明格式版本、包 ID、语义版本、兼容的运行时版本、内容项类型、Schema ID、相对路径、语义内容哈希、依赖包、入口项和变更摘要。内容项只允许 POSIX 相对路径，禁止绝对路径、反斜杠和 `..`。JSON 内容哈希基于 UTF-8 的确定性键排序序列化，不受缩进和换行影响；原始 ZIP 哈希作为独立存储完整性证据。

导入流程为：上传 → 解压隔离 → 路径与哈希检查 → Schema 校验 → 引用解析 → Context 来源白名单检查 → 静态安全检查 → 样例测试 → 生成预览 → 管理员发布。

包版本发布后不可变。多个包通过 `ContentRelease` 组合为一次可用发布；项目创建时固定 `content_release_id`。禁止“修改当前模板后所有老项目静默变化”。

### 现有 ShanHai Skills 的导入映射

管理端提供 `ShanHai Skill Bundle` 导入适配器，把现有技能包规范化为内容包，不要求运营人员手工拆文件：

| Skill 内容 | 运行时映射 |
| --- | --- |
| `SKILL.md` | `generation_template` 的职责、模式、方法和硬门禁 |
| `agents/*.yaml` | 展示名、触发语和默认入口元数据 |
| `assets/*.schema.json` | 正式输入/输出合同与动态内容定义 |
| `references/*.md` | 分阶段 handbook、rule_set 和提示词材料 |
| 注册表与上下游 | 能力图、输入输出类型和工作流引用 |
| 测试样例 | `evaluation_suite` |
| 确定性渲染规则 | `projection_template` 或已注册 renderer ID |

上传包中的 Python、Shell 或可执行文件只做隔离扫描和人工审计，不在业务运行时直接执行。若某个校验器或渲染器确实需要代码，必须作为经过评审、签名和部署的运行时扩展发布，内容包只引用已注册扩展 ID。这样既保留 Skills 的业务“血肉”，又不把管理端变成远程代码执行入口。

导入预览必须展示：将新增或替换哪些能力、Schema、提示词、规则、工作流、锚点和评测；哪些文件被忽略或隔离；与当前 Release 的破坏性差异。管理员确认后才发布。

### Schema 驱动生成模板

每个 V1 内容项使用共同信封 `api_version + kind + metadata + spec`。Manifest 负责稳定键、类型、路径、Schema 和哈希；对应类型的 JSON Schema 只校验 `spec`。跨条目引用使用结构化的 `item_key + kind`，不在导出包中暴露数据库 UUID。

一个 `GenerationTemplate` 至少固定引用：

- 一个 `InputDefinition`；
- 一个 `PromptTemplate`；
- 一个 `ContentDefinition` 输出合同；
- 零个或多个 `StylePreset`；
- 一个或多个 `ProjectionTemplate`，并标明教师视图、Provider请求或导出用途；
- 一个逻辑模型能力，不引用具体供应商模型。

后台字段通过输入定义和内容定义描述；前端只消费其中的标签、可见性和控件提示。主要字段可直接展示，次要或高级字段按需展开，系统与上下文字段保持隐藏。教师始终可以查看一段完整业务提示词或可读作品，不要求理解内部 Prompt 组件。

图片提示词可以先生成通过 Schema 的内容、构图、色系、质感、光线、风格和负面约束对象，再由 `ProjectionTemplate` 渲染为一段完整提示词。PPT设计稿可以先保存页面类型、教学目标、视觉风格、背景策略、版式、可编辑元素和资产需求，再投影为Markdown、预览和PPTX。

### Markdown模板适配层

内容管理员不直接编辑Manifest、JSON Schema、稳定引用或哈希。Markdown适配层接收普通UTF-8 Markdown，通过标准Markdown AST识别文档标题、一级章节、子章节和正文块，生成可审核的`TemplateDraft`。管理员在草稿层确认章节角色、内容来源、必填、可编辑、重复和最终展示设置；只有确认后的草稿才能编译为正式内容包。

Markdown是管理员创作、导入和预览格式，不是发布后的运行时事实源。原始Markdown作为来源证据保存；草稿可修改；正式内容包发布后不可变。Word适配器后续只产生Markdown并进入同一流程，不复制结构识别、草稿编辑或发布逻辑。

外部Markdown按不可信输入处理：限制编码和大小，拒绝原始HTML、脚本、危险链接和V1不支持的图片；文档中的指令只作为待审核业务内容，不得覆盖平台安全层或自动触发发布。确定性解析负责基础结构，AI语义建议只能辅助管理员确认。

## 4. 动态内容结构

`ContentDefinitionVersion` 用字段树表达教案、PPT 页设计、故事或分镜的结构。每个字段至少包括：

- `field_key`：稳定机器键，发布后不可改；
- `label`、`description`、`type`；
- `required`、`editable`、`deletable`、`repeatable`；
- `min_items`、`max_items`、长度或数值限制；
- `ui_widget`、占位说明、排序和折叠规则；
- `generation_instruction`、`validation_rules`；
- `export_mapping`：映射到 DOCX、PPTX、PDF 或视频脚本；
- `visibility_condition`：基于项目分支或其他字段的声明式条件。

默认教案结构为：教学内容、教材分析、学情分析、设计意图、教学目标、教学重难点及突破策略、教学准备、教学过程、板书设计、课堂总结、分层作业、教学反思。它是首发内容包默认值，不是代码常量。管理员可导入新版字段和子结构；旧项目继续使用固定版本，也可显式迁移。

三类九套拥有独立的 `intro_option_set` 内容定义，不作为教案字段树的固定子字段。教案导出器可以把已批准方案集投影为正文后的附录，但数据库、审核和版本始终分离。

### 单一语义源与双输出

每个正式内容产物先保存通过 Schema 的结构化 JSON，教师编辑器直接修改同一结构。Markdown、DOCX、PDF、PPTX 预览、视频节点文档等由受版本管理的确定性投影生成，不再让模型自由写第二份“教师版”。

投影文件记录 `source_artifact_version_id`、`projection_version_id` 和内容哈希。源 JSON 修改后，旧投影标记过期；只有重新投影且哈希检查通过才可交付。内部 ID 和枚举可转成人类标签，但正文不能被投影器摘要或改写。

## 5. 上下文绑定与隔离

每个提示词模板声明可读取的 `ContextBinding`，运行时按白名单装配上下文。允许的来源示例：

- `material.approved_parse`
- `lesson_division.approved_version`
- `lesson_plan.approved_version`
- `project.teacher_preferences`
- `asset_slot.lesson.02.video.style_master`

运行时同时校验来源已注册、调用方只提交模板声明来源、必填绑定、条目数和字节上限；任一检查失败都必须在调用模型前停止。ContextSnapshot按稳定来源版本排序并计算语义哈希，Provider不能自行补读数据库或未声明资产。

独立创意节点采用严格隔离策略：只能读取锚点库、用户通用偏好和安全规则，禁止读取 `material`、`lesson`、`knowledge_point`、`lesson_plan`。锚定节点可以读取批准课时范围和教材证据，但只能填写课程锚点、课堂首问、交接时刻、不得提前讲授内容、适配理由和推荐度，禁止改写独立创意字段。视频节点只读取被选方案快照，禁止读取完整教案。

## 6. 提示词编译

最终请求提示词按固定层次编译：

1. 平台不可修改的安全和租户规则；
2. 当前节点角色与任务边界；
3. 内容包中的业务方法和手册；
4. 动态内容结构与输出合同；
5. 白名单允许的上游批准版本和项目资产；
6. 教师选择、偏好和本轮返修意见；
7. 机器可校验的 JSON Schema；
8. 供应商适配层的格式要求。

前端展示的是第 2 至第 6 层组合后的“业务提示词”，教师可以编辑允许编辑的区块。第 1、7、8 层折叠展示说明但不可直接修改。提交生成时创建 `PromptSnapshot`，记录所有层的文本、模板版本、变量摘要、用户修改差异、上下文快照和最终请求哈希。

结构化 Prompt 组件和教师最终看到的完整提示词同时保留。教师修改完整提示词时创建不可变 `PromptRevision`，本次调用以该修订为最高优先级；系统保存基础结构、基础投影哈希和修订差异，不把自由文本反向解析回字段。基础输入、模板或投影变化后，旧修订标记为基于旧基础，必须明确沿用或重新合并。

返修默认形成新提示词修订并重新生成，不对旧模型响应做不可追溯的覆盖。对支持多轮编辑的供应商，也必须把此前结果和新指令固化到新快照。

## 7. 结构化输出与质量门槛

模型输出依次经过：

1. 供应商响应适配；
2. JSON 提取和 Schema 校验；
3. 受限的格式修复；
4. 业务规则校验；
5. 安全和敏感信息检查；
6. 内容质量评分；
7. 保存原始响应、规范化结果和校验报告。

修复不能偷偷改变业务语义。超过修复次数后，节点进入可编辑草稿或失败状态，不得把不合格内容伪装成成功。

## 8. 模型网关定位

模型网关把业务所需能力与具体供应商解耦。业务模块只请求逻辑能力，例如：

- `text.structured.zh_primary_math`
- `vision.analyze.textbook_page`
- `image.generate.education_16x9`
- `image.edit.consistent_character`
- `video.image_to_video.10s`
- `audio.tts.child_friendly_zh`

网关负责选择供应商、参数转换、限流、重试、熔断、回调、计量和统一错误；它不决定教案结构、故事方法或审核规则。

## 9. 模型配置模型

- `ModelProfileVersion`：逻辑能力、输入输出模态、参数 Schema、成本单位、质量标签和数据政策。
- `ProviderConfig`：供应商、区域、端点、密钥引用、并发限制、回调能力和启用状态。
- `ModelRoute`：逻辑能力到供应商模型的主路由、备路由、条件和权重。
- `RoutingPolicy`：按组织、节点、质量、预算、地区、文件大小和降级策略路由。

密钥只保存为 Secret Manager 的 `secret_ref`，不进入数据库明文字段、日志、导出包或前端。管理端可以测试连接、查看脱敏配置和逐步启停路由。

## 10. 路由、重试与熔断

路由顺序：能力匹配 → 租户允许 → 数据政策 → 输入规格 → 预算 → 健康度 → 质量优先级 → 权重。系统必须记录最终为什么选择该路由。

- 网络超时、5xx、限流可按策略重试或切备路由。
- 参数错误、内容安全拒绝、无余额不在同一路由盲重试。
- 熔断器按供应商模型和错误类型统计，半开探测不占用正常用户任务。
- 切换供应商时保持业务输入快照不变，只产生新的 Attempt 和供应商请求格式。

## 11. 用量、预算与审计

每次 Attempt 写 `UsageRecord`：组织、用户、项目、节点、逻辑模型、供应商、输入输出单位、计价版本、估算/实付成本、币种和延迟。模型原始响应存储遵循供应商政策和系统保留期。

Fake与真实Provider都通过ModelGateway的同一审计端口。端口只接收请求ID、能力、最终请求哈希、路由摘要、用量和稳定错误码，不接收完整提示词、模型正文、密钥或Provider原始错误；失败Attempt仍写零或已知用量记录，避免成本与失败率出现审计空洞。

预算检查分三层：组织月度限额、项目预算、单节点/单批次限额。超出预计成本时半自动和全自动模式必须暂停并让用户确认。

## 12. 管理端发布治理

内容中心和模型中心至少支持：

- 导入、校验、预览、测试、发布、停用、回滚到旧发布；
- 版本依赖和受影响项目预览；
- 只读查看线上使用的提示词与结构快照；
- 灰度到指定组织或测试项目；
- 双人复核高风险模型路由和平台安全规则；
- 完整审计：谁在何时导入、测试、发布、启停或迁移。

“回滚”是把新项目默认发布指针切回旧的不可变版本，不是修改或删除已发布内容。
