# 测试、迁移与交付

## 1. 基座使用原则

已有系统可复用的是工程基座和经过验证的后端能力，不是旧前端布局或旧业务口径。

**可复用**：认证、用户组织、文件上传、对象存储适配、基础模型客户端、任务队列、日志、部署脚本中经过验证的部分。

**必须重构**：项目/课时边界、产物版本、项目资产槽位、创作包与保存回项目、动态内容运行时、工作流状态机、模型网关路由、SSE 事件和 PostgreSQL Schema。

**不迁移**：旧前端组件树、旧页面布局、把提示词写死在代码里的实现、用 Redis 或前端代表业务进度的逻辑、把项目创作台与通用创作中心混为一体的数据结构。

旧 SQLite 或临时数据库只能作为数据样本来源，不是新系统合同。迁移脚本通过正式应用服务或明确的 ETL 写入新 Schema。

## 2. 实施阶段

本文定义设计和验收边界，不替代用户审核后的逐任务实施计划。建议交付顺序：

1. **工程基线**：模块骨架、PostgreSQL、Alembic、S3、Redis、Outbox、鉴权、可观测性。
2. **项目主干**：项目、教材上传解析、课时、动态教案、草稿/版本/审核、资产包。
3. **内容与模型**：内容包导入发布、提示词编译、模型网关、用量和预算。
4. **创作底座**：创作包、通用图片/视频创作批次、候选、保存回项目、任务面板。
5. **PPT 分支**：大纲、页设计、封面/风格、批量生图、预览、可编辑 PPTX。
6. **视频分支**：锚点隔离、故事、粗细分镜、图片、片段、音频、合成。
7. **自动化治理**：`guided/automatic`策略、节点级暂停覆盖、局部重跑、内容迁移、模型和成本管理。

每一阶段必须形成端到端可演示切片，不能只完成表和接口而没有前端可验证的用户路径。

## 3. 测试金字塔

### 领域单元测试

覆盖状态转换、权限、不变量、自动化暂停、锚点上下文隔离、字段可删/必填、精确失效、候选选择和保存回项目。领域层测试不依赖网络和数据库。

### 数据库集成测试

使用真实 PostgreSQL，覆盖唯一约束、部分索引、乐观锁、事务回滚、Outbox、租约抢占、回调去重、版本不可变和租户隔离。禁止用 SQLite 冒充 PostgreSQL 集成测试。

### 合同测试

OpenAPI 与 JSON Schema 是前后端共同合同。CI 校验：当前OpenAPI的operationId与FastAPI运行时双向一致；规划OpenAPI与当前合同严格分离；后端响应符合Schema；前端生成类型只来自当前合同；Mock场景与真实状态/错误码一致；SSE可去重和续传；内容包与动态字段符合注册Schema。

Skill Bundle 合同测试还要验证：注册表上下游可解析；正式 JSON 通过其 Schema；投影文件由当前 JSON 确定性生成；包内脚本不会被内容运行时执行；导入方案节点只获得声明的最小课程种子，不获得完整教案、PPT或视频上下文。

### 模型适配器测试

每个供应商使用录制的脱敏响应和沙箱账户测试参数映射、异步回调、轮询、超时、取消、限流、内容拒绝和计量。业务测试默认使用可控 Fake Gateway，避免非确定模型响应造成 CI 抖动。

### 端到端测试

至少覆盖：

1. 上传教材 → 批准课时 → 生成/编辑/批准教案；
2. 只做教案、教案 + PPT、教案 + 视频三种分支组合；
3. 教案未批准时可以完成三类九套和方案选择，视频上下文只包含选定方案快照；
4. 项目导出固定来源和目标的创作包 → 保存提示词版本 → 批量生成 → 采用候选 → 幂等写回固定槽位；
5. 独立创作不带项目上下文 → 保存提示词版本 → 生成并采用 → 之后显式选择有权限项目保存；
6. 上游批准新版 → 只标记真实下游过期 → 局部重做；
7. 页面刷新、SSE 断线、Worker 重启、供应商失败后的恢复；
8. `automatic`遇到预算、质量或必审点时暂停，`guided`在主要节点结果形成后暂停；
9. 生成可编辑 PPTX 和一分钟以上完整视频并形成交付包。

## 4. 内容回归评测

用脱敏教材和固定输入建立小学数学评测集，至少包含百分数、图形、计算、解决问题等类型。每次内容包或模型路由发布前检查：

- 课时划分范围完整且不越界；
- 教案结构满足当前定义，教学过程可实施；
- PPT 封面有设计、正文纯白且每页有图片主视觉，关键数学信息可编辑，页级内容与教案一致；
- 三类九套由最小课程种子直接生成最终方案，视频只消费选择快照且不读取完整教材、教案或PPT；
- 粗分镜、母图、图片资产、细分镜顺序正确；
- 镜头时长、角色/场景一致性、字幕和音频；
- 结构修复率、人工返修率、成本和延迟不明显退化。

非确定输出采用规则评分、模型评审和人工黄金样例组合，不把一次模型结果作为永久真值。

## 5. 性能与安全验证

性能至少验证普通查询、项目聚合和草稿保存 P95；100 个并发生图条目的入队与限流；SSE 千连接级心跳和恢复；50 页教材、30 页 PPT、1–5 分钟视频；大项目资产分页；Outbox 和任务表增长。

安全至少验证跨租户访问、CSRF/CORS、回调重放、恶意 Office/PDF、压缩炸弹、SSRF、对象键穿越、提示注入、锚点隔离绕过、并发双批准、重复保存和日志泄密。高风险问题不得以“后续优化”形式带入生产。

## 6. 数据迁移策略

1. 盘点旧库真实数据与质量，冻结字段映射。
2. 创建新 Schema 和稳定 ID 映射表。
3. 把旧项目映射为 Project/LessonUnit；无法判定的记录进入隔离清单，不猜测。
4. 旧文档和媒体导入 FileAssetVersion，并计算哈希去重。
5. 旧提示词和输出保存为历史 ArtifactVersion/PromptSnapshot；没有证据的批准状态不能伪造。
6. 运行完整校验、数量对账和抽样比对。
7. 灰度读新库，短期双写只用于明确边界，尽快切断旧写入。
8. 切换后保留只读备份和回退窗口，再按策略归档。

迁移脚本必须幂等、可重复运行、输出逐条错误报告和总量对账。

## 7. CI/CD 门禁

合并前：格式与静态检查、单元测试、PostgreSQL 集成测试、迁移升级/降级检查、OpenAPI 破坏性变更检查、前端类型检查/组件测试、依赖和密钥扫描。

发布前：内容回归、关键 E2E、数据库备份验证、迁移耗时评估、供应商沙箱冒烟、功能开关和回滚方案。

生产发布采用小流量灰度；数据库先做向后兼容扩展，应用切换后再清理旧列。内容包、模型路由和代码可以独立回滚，但必须记录兼容矩阵。

## 8. 完成定义

某个业务节点只有同时满足以下条件才算完成：

- 用户路径和异常路径可在前端完成；
- API/事件合同、权限和错误码实现；
- 数据模型、迁移和核心约束实现；
- 刷新、重启和重复请求后状态正确；
- 日志、指标、成本和审计可查看；
- `automatic/guided`引用同一工作流和领域数据，节点级策略覆盖行为通过测试；
- 保存提示词版本、生成、采用和写回项目四个动作的权限、幂等与审计边界通过测试；
- 文档和合同同步更新，没有另建“临时最终版”文件。
