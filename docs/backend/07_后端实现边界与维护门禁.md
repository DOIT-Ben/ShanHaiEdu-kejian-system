# 后端实现边界与维护门禁

本文件面向后端开发者、审查者和接手项目的智能体，定义当前模块所有权、事实分层和防止堆积式开发的维护门禁。它不保存任务流水；具体范围、验收和交接仍以 GitHub Issue/PR 为准。

维护责任：修改模块边界、公共合同、规模门禁或跨模块事务的任务必须同步本文件。若未来 `AGENTS.md` 和自动化门禁完整承担本文件职责，应在同一 PR 删除本文件。

## 1. 事实分层

| 问题 | 权威来源 |
| --- | --- |
| 当前产品语义 | `docs/product/`、`docs/workflows/` 和已批准 Decision Issue |
| 当前公共接口 | 已注册 FastAPI operation、`contracts/` 和消费者测试共同成立 |
| 当前数据库 | Alembic、ORM 模型和 PostgreSQL 集成测试 |
| 当前运行行为 | 代码与新鲜测试证据 |
| 未来能力 | GitHub Issue；不得混入当前设计 |
| 历史版本 | Git 历史、已关闭 Issue 和已合并 PR |

任何两层不一致时停止实现并更新 Issue。不得通过同时支持两套相反语义来掩盖冲突。

## 3. 跨模块规则

1. 一个模块不得直接调用另一个模块的内部 Repository 或绕过其授权服务写表。
2. 跨聚合事务必须有一个明确的应用编排者；它可以调用多个模块应用接口，但不能复制各模块状态机。
3. 当前 Upload 确认、Artifact stale 传播和 Creation 写回是已存在的跨模块编排边界；新增通用执行器前必须先定义同等清晰的事务所有者。
4. Worker 使用系统 Actor，但仍必须按 organization_id 和任务固定资源校验租户边界。
5. 项目/独立创作、automatic/guided 通过判别合同和策略表达，不复制表、DTO 或工作流图。
6. Provider 私有请求、模型名和密钥只存在于服务端 Adapter/Settings，不进入浏览器、内容包或教师可编辑 Prompt。

跨模块 ORM 直接导入已经集中在 Artifact、Asset、Upload、Prompt/Model Audit 编排路径。现有代码可在有测试的应用编排层保留，但新功能不得继续扩大；治理任务见 [Issue #92](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/92)。

### #89 最小运行时端口

通用节点执行器只依赖 `apps/api/runtime_boundary/ports.py` 中的消费者端口，不直接导入其他模块 ORM 模型或内部 Repository：

| 端口 | 最小职责 |
| --- | --- |
| `RuntimeDefinitionReader` | 读取项目固定 Release 对应的节点绑定和 GenerationTemplate |
| `WorkflowExecutionPort` | 锁定节点执行上下文并通过工作流状态机推进状态 |
| `ArtifactPort` | 读取批准上游版本，并在调用方事务内写入已校验 ArtifactVersion |
| `AssetPort` | 读取教材解析、参考文件版本和资产事实，不暴露 Upload/Asset ORM |
| `PromptSnapshotPort` | 通过现有 PromptSnapshotService 冻结 Prompt/Context |
| `ModelInvocationPort` | 通过 ModelGateway 和 AttemptAuditSink 发起带审计的确定性调用 |
| `CreationPackagePort` | 在调用方事务内发布固定来源与目标的不可变 CreationPackage |

Upload 的确认与写入继续由 `UploadConfirmationService` 所有；#89 只通过 `AssetPort` 读取已经确认的教材/资产事实。上述文件只定义边界和不可变 DTO，具体执行器、适配器与事务编排仍由 [Issue #89](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/89) 实现。

## 4. 状态与数据所有权

- PostgreSQL 是业务真相；Redis 中的队列、缓存和锁必须可从数据库恢复。
- AutomationPolicy 是执行模式唯一事实；项目旧 automation_mode 列不参与读取。
- WorkflowDefinition、ContentRelease、Prompt/Context 快照、ArtifactVersion 和已发布内容版本不可原地修改。
- Artifact 的 draft/submitted/approved 指针必须与状态机一致。
- Adoption 只表示选中候选；SaveToProjectOperation 才表示正式项目写回。
- FileAssetVersion 保存对象事实；短期签名 URL 每次重新签发，不持久化。
- 幂等缓存不能替代当前授权，授权检查必须先于 replay 和冲突响应。
- 普通 CI 只能调用确定性 Fake；Adapter 改动和阶段出口必须运行受控真实冒烟。

## 5. 公共合同门禁

一个操作只有同时满足以下条件，才能称为当前可用 API：

1. FastAPI 已注册唯一 operationId。
2. 权威 OpenAPI 描述请求、响应、错误和必要响应头。
3. 生成客户端已更新。
4. Mock/消费者合同通过。
5. 至少有一个真实路由或纵向集成测试。

仅存在于 OpenAPI 的规划操作不得进入当前生成客户端。当前静态/运行时操作分层由 [Issue #85](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/85) 收口。

Cookie 会话、CSRF、CORS 和反向代理可信边界属于部署合同，不得依赖默认值猜测；见 [Issue #84](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/84)。

## 6. 规模与复杂度门禁

仓库级触发线沿用 `AGENTS.md`：

- 生产源文件超过 400 行：拆分或在 PR 解释例外。
- 函数超过 60 行：拆分或说明不可分原因。
- 单 PR 超过 20 个业务源文件或净新增 800 行：提供 review map，并说明不能继续拆分的原因。
- 同一业务逻辑第三次重复：抽取到拥有该事实的模块，或在评审中说明保留原因。

当前已经触发文件或函数阈值的压力点：

| 文件 | 当前信号 | 后续要求 |
| --- | --- | --- |
| `uploads/confirmation_service.py` | 422 行；上传、对象复制、资产、Job 和事件编排集中 | 新增职责前先提取边界 |
| `artifacts/router.py` | 422 行；四类命令、序列化和 ETag 集中 | 新端点进入独立路由或 presenter |
| `assets/project_service.py` | 419 行；槽位声明、绑定、替换、解绑和验证集中 | 保持应用接口，拆出独立校验职责 |
| `model_gateway/gateway.py` | 410 行；文本、图片与视频调用编排集中 | 保持网关接口，按独立任务拆分多模态编排 |
| `artifacts/approval_service.py` | 审核状态机和 stale 传播集中 | 不再加入无关 Artifact 命令 |
| `artifacts/service.py` | 草稿、提交、关系入口集中 | 通用执行器通过接口调用，不内嵌 |

当前确定性扫描发现 4 个生产文件超过 400 行、23 个生产函数超过 60 行，其中 `creation/save_service.py::save` 为 123 行。相较 Issue #92 创建时记录的 3/21，实时扫描额外纳入了 `model_gateway/gateway.py`、CLI 函数和嵌套函数；这反映扫描口径完整化，不代表本任务新增业务增长。Worker 解析、PDF 解析、Prompt 编译、模型网关和 AutomationPolicy 更新也已超过门禁。它们是有归属的存量风险，不是继续增长的许可；机器基线位于 `scripts/repository-governance-baseline.json`，每项均记录 owner、理由和退出 Issue。新任务不得增加这些超限项的行数或引入未登记超限项。

规模扫描覆盖 `apps/api/**/*.py` 与 `workers/**/*.py`，文件使用物理行数，函数使用 AST 的 `lineno` 至 `end_lineno` 并包含嵌套函数。跨模块门禁解析绝对、相对和包级 alias 导入，允许清单精确到来源文件、目标 ORM 模块和导入符号集合；新增或扩大的直接模型导入均失败。`apps/api/model_registry.py` 是 SQLAlchemy 映射注册的组合根，其包级 ORM 导入也逐项登记，不使用目录级豁免。

当前搜索未发现无归属 TODO/FIXME/HACK。未来临时开关和 TODO 必须链接移除 Issue 与目标里程碑。

## 7. 审查清单

- 是否从正确模块的应用接口访问数据？
- 是否新增第二 DTO、第二状态机、第二枚举或第二默认值？
- 是否在 replay 前重新执行当前授权？
- 是否把计划能力误写成当前事实？
- 是否为数据库变化提供 Alembic、空库升级和约束测试？
- 是否为 OpenAPI 变化重新生成客户端并运行消费者合同？
- 是否保持 Fake/真实 Provider 门禁分离？
- 是否删除同一变更中失效的代码、合同和文档？
- 是否触发文件、函数或 PR 规模解释？
