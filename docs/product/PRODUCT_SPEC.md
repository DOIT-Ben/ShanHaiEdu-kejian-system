# 产品规格

状态：当前唯一产品基线

## 1. 产品定位

山海教育课件系统是基于标准课件生产工作流的AI一站式创作平台，不是臃肿的流程控制台，也不是多个模型网站的跳转集合。系统把经过设计的课件生产方法固化为一条工作流：教师既可以让系统端到端推进，也可以进入任意主要环节修改和确认，最终成果始终沿同一流程生产。

系统价值来自三部分：

- 完整课件项目：产品主入口，负责教材、课时、教案、PPT、课堂导入视频和交付。
- 通用创作台：项目节点的专业执行环境，也是次级独立图片、视频和PPT创作入口。
- 平台能力：内容包、工作流、Prompt编译、模型网关、资产版本、自动化和审计。

## 2. 用户和成功标准

第一阶段主要用户是小学数学教师。内容管理员维护教案结构、Prompt、规则、锚点和工作手册；模型管理员维护模型服务；系统管理员维护权限和审计。

教师应当能够：

- 上传一个小知识点的教材PDF；
- 选择一键生成整套课件或按步骤辅助创作，并在同一项目中切换参与程度；
- 获得可编辑课时划分；
- 为每个课时生成和编辑教案；
- 自由选择是否制作PPT和课堂导入视频；
- 在生成前看到并修改最终业务生成指令；
- 查看当前作品、候选结果、历史版本和费用；
- 从项目一键进入创作中心；
- 在项目创作中把已采用结果写回创作包固定槽位；在独立创作中显式选择有权限项目和槽位附加；
- 在失败、刷新、离开页面和重试后继续工作；
- 下载教案、PPTX和完整视频。

## 3. 项目边界

一个`Project`对应一个小知识点教材，例如“认识百分数”。项目内包含一个或多个`LessonUnit`。课时不是独立项目。

每个课时拥有：

- 必选教案；
- 默认生成、可关闭的三类九套课堂导入设计附录；
- 可选PPT分支；
- 可选课堂导入视频分支；
- 中间作品、素材、版本和最终交付物。

教案正文和课堂导入设计是独立审核的兄弟产物，不是“先有教案再从教案提炼视频主题”。导出教案时可把三类九套作为正文后的附录。PPT读取批准教案；视频只读取教师选中的单个导入方案快照，不读取教案正文。

分支`disabled`表示未启用；`skipped`表示本次运行有意跳过。最终交付只检查已启用分支。

## 4. 一个工作流内核与两种执行方式

- **一键生成 `automatic`**：系统在授权、预算、质量和门禁范围内自动运行可执行节点；只在高价值人工门禁、费用阈值、硬规则、无法自动修复的质量问题或失败时暂停。
- **按步骤辅助创作 `guided`**：系统仍先按标准方法准备内容和业务生成指令，但在主要节点生成后暂停，教师可以查看、修改、确认再进入下一步。

两种执行方式引用同一个`WorkflowDefinition`，共用项目、节点运行、任务、版本、依赖、质量门禁和资产；差异只来自版本化`AutomationPolicy`中的自动启动、自动采用、自动写回和人工暂停规则。教师可以在允许范围内调整参与节点，这属于策略覆盖，不形成第三套“半自动”产品模式，也不复制工作流或数据。

现有`manual/assisted/automatic`只作为迁移输入：`manual`迁为`guided`且节点自动动作全关，`assisted`迁为`guided`并保留原节点规则，`automatic`保持不变。已运行项目继续保留原始策略快照用于审计，不原地改写历史；公开合同和数据库按扩展、兼容读取、回填、切换、清理的顺序迁移。

## 5. 动态内容结构

系统内置小学数学教案默认十二部分：教学内容、教材分析、学情分析、设计意图、教学目标、教学重难点及突破策略、教学准备、教学过程、板书设计、课堂总结、分层作业和教学反思。

十二部分只是默认内容定义，不是程序常量。管理员可以导入新的章节、嵌套字段、必填级别、质量规则、UI和导出映射。前端通用渲染，后端固定内容定义版本。

管理员把十二部分调整为十部分时必须发布新的内容定义版本。新项目可以使用十部分，旧项目继续固定十二部分；旧项目迁移前必须展示字段新增、删除、合并和潜在内容损失，不允许静默重写已有教案。

内容管理员的日常主入口是Markdown模板适配器，而不是手写内容包合同。管理员上传或粘贴普通Markdown后，系统识别标题、章节、子章节和正文结构，形成可调整但不可直接发布的模板草稿；管理员确认内容来源、必填、可编辑、重复和展示设置后，系统再编译为结构化内容定义和确定性教师投影。Word导入后续只负责先转换为Markdown，不建立第二套模板构建流程。

外部Markdown始终作为不可信输入，不得直接覆盖平台安全层、执行HTML或脚本，也不得绕过管理员确认。原始Markdown用于来源追溯，模板草稿可修改，已发布内容版本不可变。

模板草稿只有在管理员确认并进入`ready`后才能编译。发布配置与导入正文分离：受Schema约束的CompilationProfile声明版本、逻辑模型能力、白名单上下文和教师可见业务指令；导入正文只能成为字段默认内容、教师初始输入或字段级生成说明，不能变成隐藏系统指令。相同草稿和发布配置必须得到字节一致的内容包，结构调整必须发布新语义版本。

三类九套使用独立内容定义和产物版本，默认随教案展示但不嵌入十二部分字段树。教师未选择某套方案不妨碍教案正文批准；选择结果属于项目运行状态。

默认路径生成科普、应用、故事各三套；教师可以关闭导入设计。教师已有明确创意时仍使用同一个`intro_option_set`产物族，以`generation_mode=refine_existing`绑定一个同课时exact来源版本并只完善一套，不建立第二条工作流。方案集通过exact质量报告后先独立批准，选择只能引用该已批准版本。教师可以选择任意方案；只有教师未选择且AutomationPolicy对`intro.select`显式允许`auto_select`时才采用唯一最高分，选择记录必须区分`teacher_selected`与`policy_default`。

已发布内容定义中的字段结构不可由教师删除；`deletable`描述字段结构是否可从模板草稿中移除，不等于清空字段值或删除`repeatable`条目实例。教师可以修改可编辑值，也可以在内容定义允许的数量范围内增删重复项；管理员改变章节或字段结构时必须发布新内容定义版本。

### Schema 驱动生成模板

输入和输出都可以由内容管理员模板化，但后台结构化不等于教师端展示专业字段。一个可执行 `GenerationTemplate` 组合：

- `InputDefinition`：教师填写、系统补全、上下文注入和隐藏字段；
- `StylePreset`：图片、视频和PPT可选的视觉语言及模态扩展；
- `PromptTemplate`：把输入、白名单上下文、业务方法和输出合同编译为业务提示词；
- `ContentDefinition`：模型必须返回的结构化字段；
- `ProjectionTemplate`：把结构化结果确定性渲染为完整提示词、Markdown、DOCX、PPTX或其他教师可读形式。

例如图片提示词在后台可以包含内容、构图、色系、质感、光线、风格和负面约束，教师端仍默认展示一段完整可编辑提示词。教师修改完整提示词时保存本次 `PromptRevision`，系统同时保留原结构化对象和修改差异，不尝试把自然语言可靠反向解析回字段。

PPT设计稿的后台事实包含页面类型、教学目标、视觉风格、背景策略、版式、可编辑元素和图片资产需求；教师看到的Markdown设计稿、页面预览和PPTX都由同一结构化事实投影，不维护相互独立的第二份文档。

#130、#146与#116形成的不可变`1.1.0`继续保持原合同；#127与Decision #185以新的`shanhai.primary_math.courseware@1.2.0`前向候选修正Intro同一Artifact返修血缘、workflow gate批准终态和质量支持输入。v2目录仍为48节点，内容包仍为22个模型节点分别提供输入定义、业务Prompt、输出定义、教师投影和生成模板，并使用“1～5的认识”脱敏黄金项目验证课时、十二部分教案、三类九套、PPT、图片、视频和音频的跨成果边界。显式PostgreSQL发布会把`1.2.0`设为后续新项目默认；这不等于生产环境已执行发布，现行正式`1.0.0`、不可变`1.1.0`和既有项目绑定均保持不变。黄金项目只固定结构、来源、上下文和质量不变量，不把一次模型自然语言响应当作永久逐字真值，也不代表真实Provider或媒体已经完成。

课时划分节点只输出课时边界和教材覆盖，不连带生成教案。教案和课程驱动导入方案分别使用独立模板；PPT读取批准教案和教材证据，视频模板只读取已选导入方案快照。业务实现可以在这些合同合并后按教案、PPT和视频独立Issue推进，但不能复制字段、状态机或Provider私有参数。

PPT页数和视频时长采用[Decision #73](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/73)批准的“系统推荐、教师覆盖”口径。PPT期望页数可选，平台接受5至60页，系统通常根据教学任务推荐10至20页；视频根据母版故事的场次、节拍、镜头数和交接点推荐60至180秒。实时费用由服务端根据当前路由与版本化价格档案估算，不由模型或内容包生成。教师确认视频时长与费用后才能批准母版并进入粗分镜。

TTS属于后续能力。当前内容包可以保留旁白、对白、声音意图、字幕和时间线等音频计划数据，但不要求TTS Provider、真实TTS冒烟或音频文件生成，也不以TTS阻塞当前PPT和视频合同出口。

## 6. 项目与创作中心

图片、视频和PPT创作台是平台级通用工具。项目工作流运行到专业制作节点时，系统带入完整上下文和固定保存目标；用户也可以从次级入口独立使用。

```text
WorkflowRun / NodeRun
→ 创建不可变CreationPackage
→ 导入CreationBatch / CreationItem
→ 保存业务Prompt版本并冻结PromptSnapshot
→ 提交GenerationJob并生成Candidate
→ Adopt采用候选
→ SaveToProjectOperation写回固定AssetSlot
→ 解锁下游或传播stale
```

项目批次必须携带`project_id`、`workflow_run_id`、`source_node_run_id`、`creation_package_id`和`target_slot`，不能改写来源项目或目标槽位。独立批次不伪造这些字段，结果可以下载或保存到个人资料；附加到项目时必须显式选择有权限项目与槽位并重新鉴权。

`save_prompt_version`、`generate`、`adopt`和`save_to_project`是四个独立动作。采用候选不等于写回项目，保存提示词版本不触发生成，写回项目不得依据最近访问项目猜测目标。未正式写回的候选不进入项目素材与成果。

## 7. 课程导入锚点

三类默认创意为科普、应用和故事，每类三套。锚点定义表达机制，而不是强制制造冲突：

- 科普：现象、未知、认知惊奇或解释欲望；
- 应用：真实任务、选择、需要或解决目标；
- 故事：人物目标、阻碍、变化或悬念。

系统用单个课程驱动节点读取最小课程种子，在一次生成中直接形成最终一套或三类九套方案。科普、应用和故事是主要倾向，辅助倾向允许交叉；每套同时形成课程关联、课堂首问、交接时刻、不得提前讲授和推荐依据。方案集先校验并独立批准，教师或显式`auto_select`策略再从exact批准版本形成不可变选择快照；视频只接收该快照。完整定义见 [三类九套课堂导入设计](../workflows/INTRO_ANCHORS.md)。

## 8. 产品体验原则

- 主页是品牌展示、创作起点和成果展厅，不是数据驾驶舱。
- 当前功能页以作品为视觉中心，默认占主要空间。
- 用户一眼看懂正在做什么、看哪里、点哪里和改哪里。
- 一个视觉区域只保留一个主操作。
- 普通教师端减少Prompt、Provider、shot、clip等专业词；只提供逻辑质量/速度档位，不默认暴露真实Provider名称、私有参数和路由细节。
- 直接在作品上编辑；高级设置按需展开。
- 工作流退居幕后，但版本、审计、费用和来源完整保留。

## 9. 非目标

第一阶段不做整册教材自动切分、实时多人协同、移动端完整创作、支付结算、开放开发者平台、复杂模型竞价和微服务拆分。
