# 需求分析

## 1. 背景与问题

小学数学教师当前能用对话模型、生图模型、视频模型和剪辑工具分别完成教案、PPT、图片和视频，但需要在多个网站间复制教材、提示词和结果。中间状态依赖人工文件夹与记忆，常见问题是：

- 同一教材被反复上传，课时、教案、PPT 和视频上下文无法稳定传递；
- 提示词只能手工复制，系统默认能力与教师个人修改不能同时保留；
- 生图、视频任务跨平台运行，候选与正式资产混在一起；
- 修改上游后只能整条重做，无法判断真正受影响的页面或镜头；
- 自动化容易隐藏输入和错误，纯人工又需要频繁切换工具；
- 模板、工作流、模型路由和业务规则散在个人文档中，无法统一发布和审计。

本系统要把这些可人工完成的步骤编排进一个平台，并把已有方法、提示词、Skills、结构和质量门变成可导入、可版本化的业务内容。

## 2. 业务目标

1. 教师只上传一次单知识点教材，在同一项目中完成课时、教案、PPT、课堂导入视频和交付。
2. 教师无需理解底层模型供应商，但能查看、修改和审核每次业务生成指令与输出。
3. 图片、视频和 PPT 创作能力可独立使用，也能与项目双向交换结构化任务和正式资产。
4. 手动、半自动、全自动共用同一数据与状态机，随时可暂停、人工接管和局部返修。
5. 平台方法能够通过标准内容包导入、测试、发布、灰度和回滚，不依赖改代码更新提示词。
6. 所有项目资产、中间版本、模型调用、审核和费用可追溯、可恢复、可交付。

首期成功不是“能调通若干模型”，而是教师能真实闭环完成一套小学数学课件，并在刷新、失败、返修和跨创作中心操作后保持状态正确。

## 3. 用户与职责

| 角色 | 主要目标 | 关键权限 |
| --- | --- | --- |
| 教师/项目Owner | 完成课件、控制质量和成本 | 创建项目、编辑、生成、审核、交付、成员管理 |
| 协作编辑 | 帮助制作内容 | 编辑草稿、提交生成、提交审核 |
| 审核者 | 确认教学和视觉质量 | 查看来源、批准、退回、接受过期内容 |
| 内容管理员 | 维护平台业务“血肉” | 导入/测试/发布结构、提示词、规则、锚点、Skills |
| 工作流管理员 | 维护节点和自动化 | 发布工作流、门禁、重试和上下文白名单 |
| 模型管理员 | 维护模型服务 | 配置逻辑模型、供应商、路由、限流和健康检查 |
| 系统/审计管理员 | 维护组织安全 | 用户、权限、预算、审计和保留策略 |

## 4. 范围

### 首期范围

- 小学数学、单个知识点教材 PDF；
- 教材上传、解析、范围确认和课时划分；
- 动态结构教案，默认十二部分；
- 每课时三类九套课堂导入设计附录；
- 课堂 PPT 大纲、页级设计、图片、预览和可编辑 PPTX；
- 课堂导入视频母版剧本、粗细分镜、图片、片段、音频和合成；
- 通用图片、视频、PPT 创作中心；
- 项目素材与成果、版本、任务、审核和交付；
- 内容包/Skill Bundle、工作流和模型服务管理；
- 手动、半自动和全自动模式。

### 不在首期

整册自动切分、多学科完整内容库、实时多人共同编辑、移动端完整制作、支付结算、开放开发者平台、复杂模型竞价和微服务拆分。

## 5. 核心业务对象

| 对象 | 需求语义 |
| --- | --- |
| Project | 一个小知识点的完整生产空间 |
| LessonUnit | 项目拆出的一个课时 |
| Artifact / Version | 可编辑作品和不可变历史版本 |
| IntroOptionSet / Selection | 三类九套方案集与独立选择状态 |
| ProjectAssetSlot | 项目需要某类正式素材的稳定位置 |
| CreationPackage | 项目导入通用创作中心的不可变任务包 |
| CreationBatch | 可独立使用、可编辑、可批量生成的创作会话 |
| GenerationJob / Result | 模型任务和候选结果 |
| SaveToProjectOperation | 候选成为项目正式资产的原子操作 |
| ContentRelease | 当前项目固定使用的结构、提示词、规则和工作流组合 |
| ModelProfile / Route | 业务能力与具体模型供应商的解耦配置 |

## 6. 关键用户旅程

### 创建一套课件

教师创建项目 → 上传单知识点教材 → 确认解析范围 → 生成并调整课时 → 为每课时生成/编辑/批准教案 → 选择 PPT 和/或视频 → 逐节点审核 → 生成交付包。

### 通过项目批量创作图片

项目节点展示待生成图片及完整提示词 → 教师确认并导出不可变创作包 → 通用图片创作中心建立批次 → 教师可改提示词/参考图 → 并发生成候选 → 选中结果 → 保存到指定项目槽位 → 项目节点恢复并显示正式资产。

### 独立使用创作中心

教师不进入项目直接生图或视频 → 保存批次和候选 → 下载，或稍后选择有权限项目和目标位置保存。未保存结果不污染任何项目。

### 局部返修

教师修改并批准一个上游版本 → 系统只标记实际引用旧版本的页面/镜头过期 → 教师选择局部重做、继续沿用或调整依赖 → 未受影响内容保持批准状态。

## 7. 功能需求

| 编号 | 需求 | 验收摘要 |
| --- | --- | --- |
| FR-01 | 单知识点项目与多课时 | 一个项目可管理多个有序课时，课时不是独立项目 |
| FR-02 | 教材证据 | 上传、解析、页段引用和范围确认可追溯 |
| FR-03 | 课时划分 | 只划分课时，不同时生成详细教案；结果可增删排序和批准 |
| FR-04 | 动态教案 | 默认十二部分；字段、子结构、必填、可删和导出由内容定义控制 |
| FR-05 | 三类九套 | 独立创意与课程锚定隔离；方案集与教案正文分版本，选择不阻塞教案 |
| FR-06 | PPT | 封面先审核；正文纯白且每页图片主视觉；关键文字/数学图形可编辑 |
| FR-07 | 视频 | 只消费选中方案；按母版剧本→粗分镜→母图→图片→细分镜→clip→合成 |
| FR-08 | 提示词面板 | 展示系统组装后的业务提示词，允许教师修改并保存修订 |
| FR-09 | 输出面板 | 当前节点输出、候选、校验、来源和返修始终可见 |
| FR-10 | 通用创作中心 | 图片/视频/PPT 可脱离项目使用；单项和批量采用同一组件 |
| FR-11 | 项目双向通信 | 项目导出 CreationPackage；结果通过明确目标原子保存回项目 |
| FR-12 | 资产与版本 | 中间产物和最终交付均版本化；候选与正式资产严格区分 |
| FR-13 | 自动化 | 三种模式同一工作流；预算、质量、风险或必审点自动暂停 |
| FR-14 | 内容运行时 | 内容包与 Skill Bundle 可导入、校验、试运行、发布和回滚 |
| FR-15 | 模型网关 | 所有模型服务端调用，可配置主备路由、限流、计量和密钥引用 |
| FR-16 | 可靠任务 | 长任务可取消/重试/恢复；刷新和SSE断线不丢失 |
| FR-17 | 权限审计 | 项目、管理端、跨项目保存和高风险发布均鉴权并审计 |
| FR-18 | 交付 | 按启用分支输出教案、PPTX、视频及质量/文件清单 |

## 8. 体验需求

- 首页负责吸引和开始创作，不是后台数据大屏。
- 功能页以作品为视觉中心，默认占主要空间。
- 用户三秒内知道做什么、看哪里、点哪里、改哪里。
- 一个视觉区域只有一个主操作，高级设置默认收起。
- 教师端使用“作品、待生成内容、画面风格、已采用片段”等语言，隐藏不必要内部术语。
- 编辑尽量直接发生在教案、页面、故事、分镜和图片上。
- 所有关键状态都有加载、空、错误、部分成功、过期、暂停、取消和权限反馈。

## 9. 数据与业务规则

1. PostgreSQL 是业务真相；Redis、SSE 和浏览器状态都可重建。
2. 已批准产物不可原地覆盖；生成和人工修改形成新版本。
3. 每次模型调用保存提示词、上下文、模型路由、参数、结果和用量快照。
4. 项目只固定引用已发布内容与工作流版本，管理员更新不静默影响旧项目。
5. 创作候选保存成功前不属于项目；保存操作必须幂等、鉴权和可审计。
6. 视频候选只有 result_id，选中且保存后才形成正式 clip_id。
7. 教案结构、三类九套、PPT页合同、视频shot合同和错误事件必须通过 Schema。
8. 上游草稿变化不影响下游；只有新批准版本触发精准过期。

## 10. 非功能需求

| 类别 | 要求 |
| --- | --- |
| 安全 | 租户隔离、最小权限、私有存储、短签名URL、密钥托管、上传扫描、SSRF/提示注入防护 |
| 可靠 | 幂等、租约、回调去重、Outbox、可恢复任务、数据库备份与对象存储一致保留 |
| 性能 | 同步API快速返回；长任务异步；大列表游标分页；生图和视频按队列隔离 |
| 可观测 | 请求、工作流、节点、任务、供应商调用和成本全链路追踪 |
| 可访问 | 教师核心路径键盘可用、焦点清晰、对比度与表单错误满足WCAG 2.1 AA |
| 可维护 | 模块边界、注册表扩展、生成类型、迁移和合同测试，禁止巨型条件与重复DTO |
| 可扩展 | 业务内容、工作流、字段、模型路由和创作类型通过版本化定义扩展 |

## 11. 风险与控制

- **模型不稳定**：结构化合同、修复上限、质量门、候选比较和人工审核。
- **费用失控**：生成前预估、组织/项目/节点预算、并发和自动化暂停。
- **上下文污染**：节点白名单、双快照、锚点隔离测试和提示词预览。
- **视觉不一致**：封面/母图先确认、风格合同、一致性键和资产版本。
- **外包前端偏离**：现行文档、机器合同、Mock 场景、组件/页面验收和预览门禁。
- **内容包执行风险**：声明式导入；脚本隔离；可信运行时扩展独立签名发布。
- **版本混乱**：当前树只保留一套稳定路径，历史只在 Git 中追溯。
