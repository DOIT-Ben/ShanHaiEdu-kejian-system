# 山海教育前端产品设计稿

状态：当前前端优化基线
适用对象：产品经理、UI/UX 设计师、前端开发、交互验收人员
产品范围：教师端项目、课时工作台、通用创作中心，以及与它们直接相关的成果、任务和资产页面

## 1. 产品定位

山海教育不是模型选择器，也不是工程任务控制台。它是小学数学教师把一份教材材料变成可直接使用的课堂作品的创作台。

教师真正关心的是：

- 我正在做哪一个项目、哪一课时；
- 当前作品是什么，是否达到了课堂可用标准；
- 下一步只需要确认或修改什么；
- 修改后的图片、视频、PPT 或教案会保存到哪里；
- 失败、等待和已完成的任务如何继续处理。

页面应把内部的工作流、Prompt、Provider、Job、Artifact 等技术概念转换成教师能理解的语言。内部结构可以严格保存，用户界面只呈现当前决定所需的信息。

## 2. 设计目标

### 2.1 核心目标

1. 教师打开首页后 3 秒内知道当前项目、当前课时和唯一主操作。
2. 教师在工作台中始终看见当前作品，不因切换节点而整页刷新、回到顶部或丢失滚动位置。
3. 图片、视频、PPT 创作统一为“上方看作品、下方输入要求”的对话式创作体验。
4. 项目上下文、提示词、参考图、比例、风格、任务队列和保存目标自动继承，不让教师重复选择。
5. 教案、母版剧本、分镜和设计稿以完整可读的文档或作品呈现，结构化字段只作为编辑和存储方式，不直接暴露给普通用户。
6. 每个页面只有一个主要决策和一个主按钮，其余能力按需展开。

### 2.2 非目标

- 不在浏览器中直接调用模型供应商或保存供应商密钥。
- 不新增第二套业务数据源或第二套 DTO。
- 不把管理端的工作流编排、模型路由、审计字段展示给教师端。
- 不把未接入真实 API 的 Mock 结果描述成生产闭环。
- 不在本轮关闭或删除 PPT Issue #106、#107、#108，它们继续作为后续 PPT 改造依据。

## 3. 用户与场景

### 3.1 主要用户：小学数学教师

使用环境以桌面浏览器为主，常见宽度为 1280、1440 和 1920；备课时也会使用 375 至 414 像素的移动设备查看、审核和下载。教师不需要知道模型名称、任务 ID 或状态机，只需要知道作品是否可用、要不要重做以及下一步是什么。

### 3.2 三个高频场景

#### 场景 A：从教材开始制作一套课件

创建项目 -> 上传知识点教材 -> 确认解析范围 -> 调整课时 -> 生成并编辑教案 -> 选择导入设计 -> 制作 PPT 或课堂视频 -> 审核 -> 交付。

#### 场景 B：从项目资产制作图片或视频

项目节点产生待制作内容 -> 自动继承项目、课时和提示词 -> 进入创作中心 -> 修改要求或导入资产包 -> 生成候选 -> 采用 -> 保存回项目资产位。

#### 场景 C：独立快速创作

从首页进入创作中心 -> 选择图片、视频或 PPT -> 输入一句自然语言要求 -> 生成候选 -> 继续对话修改 -> 下载或选择项目保存。

## 4. 信息架构

### 4.1 全局导航

桌面端只保留一条顶部全局导航：

`山海教育 | 首页 | 项目 | 创作中心 | 任务中心 | 搜索 | 通知 | 头像菜单`

管理端入口只出现在有权限的头像菜单中。项目工作台不得再叠加一套全局侧栏。

### 4.2 页面层级

```text
首页
├── 当前项目与继续制作
├── 待确认事项
├── 最近成果
└── 快速进入创作中心

项目
├── 项目列表
├── 项目概览
│   ├── 教材与课时
│   ├── 课时工作台
│   ├── 素材与成果
│   ├── 项目任务
│   └── 项目交付
└── 结果预览与版本

创作中心
├── 创作首页
├── 图片创作
├── 视频创作
└── PPT 创作
```

### 4.3 URL 与实现边界

真实 Runtime 路由以 `apps/web/docs/ROUTES_AND_API.md` 为准。Mock 路由只用于视觉和交互验证，不代表真实 API 已经接通。新增页面必须先确认它属于 Runtime、Mock 还是两者共有，禁止在同一个路由内静默切换数据来源。

## 5. 全局页面骨架

### 5.1 通用区域

```text
┌──────────────────────────────────────────────────────────────┐
│ 顶部导航：品牌 / 当前入口 / 搜索 / 通知 / 账户                │
├──────────────────────────────────────────────────────────────┤
│ 页面标题与上下文：项目名 · 课时名 · 当前状态                  │
├──────────────────────────────────────────────────────────────┤
│ 主内容区：作品、文档、列表或工作台                            │
└──────────────────────────────────────────────────────────────┘
```

页面只能有一个 `<main>`。主要区域使用 `<section>` 和明确的 `aria-label`。页面标题保留唯一 `h1`，文档内部标题从 `h2` 开始。

### 5.2 空间原则

- 主作品或文档默认占可用宽度的 70% 以上。
- 任何固定栏都不得遮挡作品和底部主操作。
- 普通页面优先使用背景层级、分隔线和留白，不让每个区域都变成浮卡。
- 普通控件圆角 6px，卡片和弹窗 8px；禁止继续增加大圆角档位。
- 100% 浏览器缩放下，页面标题、主作品和主操作应同时可见；必要内容通过页面滚动，而不是把主体缩成缩略图。

## 6. 首页设计

首页是教师的今日工作台，不是宣传页，也不是数据驾驶舱。

### 6.1 首屏顺序

```text
当前项目与课时
  ↓
下一件要完成的事（唯一主操作）
  ↓
待确认或失败任务
  ↓
继续制作
  ↓
最近成果
```

首屏内不出现“当前要做什么”“下一步说明”“导入课堂设计”等重复解释。把这些内容压缩为主操作旁的一句短状态或 Tooltip。

### 6.2 首屏组件

- `CurrentWorkCard`：展示项目名、课时名、当前节点、最近更新时间。
- `PrimaryAction`：根据状态显示“继续制作”“确认教案”“查看失败任务”等唯一主操作。
- `AttentionList`：只展示可行动事项，每行包含对象、原因、主操作和状态语义。
- `RecentWorks`：展示图片、PPT、视频或教案的真实缩略图；缩略图是导航，不承担细节检查。

品牌视觉只作为辅助背景或作品缩略图，不挤压任务区，不使用巨型 Hero 抢占首屏。

## 7. 项目列表与概览

### 7.1 项目列表

桌面端采用紧凑行卡或表格式卡片，每行展示：

`课题 | 当前课时 | 当前状态 | 下一步 | 更新时间 | 操作`

一屏应能扫描 6 至 8 个项目。搜索和“创建项目”固定在同一工具栏，不额外制造页面层级。状态同时使用图标、文案和语义色，不能只依赖浅色背景。

### 7.2 项目概览

项目概览回答四个问题：

1. 这是什么项目；
2. 有哪些课时；
3. 当前哪些成果可用、哪些待确认；
4. 从哪里继续。

项目资产和创作中心采用双向连接：从项目进入创作中心时自动带入项目、课时、资产位和目标保存位置；从独立创作中心保存时才允许选择有权限的项目。

## 8. 课时工作台

### 8.1 桌面布局

```text
┌──────────────┬───────────────────────────────────────────────┐
│ 流程导航     │ 操作栏：项目 · 课时 · 当前步骤 · 唯一主操作     │
│              ├───────────────────────────────────────────────┤
│ 1 准备课程   │                                               │
│ 2 写教案     │ 主作品区：文档、PPT、画面、播放器或成果预览     │
│ 3 选课堂导入 │                                               │
│ 4 做 PPT     │                                               │
│ 5 做视频     │                                               │
│ 6 交付       │                                               │
└──────────────┴───────────────────────────────────────────────┘
```

左侧流程导航是定位工具，不是刷新式分页。点击相邻节点时，内容在同一个工作台容器内切换，保持侧栏 `scrollTop`、页面滚动位置和局部编辑状态。只在目标节点离开可视安全区时平滑聚焦，禁止每次切换自动滚到页面顶部。

### 8.2 手机布局

顶部始终显示“项目名 · 当前课时 · 当前步骤”。流程导航折叠为水平滑动或底部流程按钮；主内容占满宽度；右侧抽屉改为底部抽屉；主操作固定在可见区域但不得遮挡正文。

### 8.3 操作栏

顶部只保留：返回、项目/课时上下文、当前步骤、一个主操作。暂停、版本、费用、来源和高级设置进入菜单或抽屉。按钮文字必须说明结果，例如“确认并继续”“保存到项目”“采用这张图”，不使用模糊的“下一步”替代具体动作。

## 9. 内容与文档呈现

### 9.1 完整故事优先

母版剧本、教案、课堂导入方案、分镜和设计稿必须以完整可读的文档或作品呈现。内部仍可存储结构化字段，但普通用户默认看到连续内容、标题、段落、列表、表格和必要的媒体。

### 9.2 编辑与预览

采用“编辑 / 预览”双状态，而不是把字段拆成一排排表单：

- 预览状态：内容优先，保留标题、段落和媒体关系。
- 编辑状态：允许直接修改 Markdown 或富文本；标题层级、列表和媒体引用可视化。
- 保存状态：显示“已保存”“保存中”“保存失败”，不展示内部版本号和 Provider ID。
- 修改后：明确提示哪些下游作品会受影响，并给出“继续沿用旧版”或“重新制作受影响内容”。

## 10. 通用创作中心

创作中心是图片、视频、PPT 共用的一种交互模型，不是三套独立后台。

### 10.1 基础布局

```text
┌──────────────────────────────────────────────────────────────┐
│ 创作类型 / 当前作品名 / 项目归属 / 保存状态                    │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│                    作品对话展示区                            │
│        AI 生成结果在左，用户每次要求在右                      │
│        图片、PPT 页面、视频关键帧以消息形式连续呈现            │
│                                                              │
├──────────────────────────────────────────────────────────────┤
│ 参考图  参数摘要  比例  风格  时长  数量       [发送/生成]      │
│ 可编辑的提示词输入框                              ↑            │
└──────────────────────────────────────────────────────────────┘
```

创作结果不能被固定在狭窄卡片里，也不能被超大单图挤出首屏。主预览桌面端建议 480 至 720px，移动端不超过 `min(100%, 360px)`；PPT 使用 16:9 画布；视频展示播放器和关键帧，不把首屏变成一张海报。

### 10.2 输入区

输入区固定在当前可视区域底部，支持：

- 多行提示词输入；
- Enter 发送，Shift+Enter 换行；
- 输入为空时禁用发送；
- 输入后右下角出现箭头发送按钮；
- 上传参考图或从项目资产导入；
- 参数摘要显示当前值，例如“课堂插画 · 1:1 · 纸艺微缩 · 3 张”；
- 点击摘要打开底部抽屉编辑，而不是隐藏成没有状态的图标。

生成中保留按钮尺寸，按钮显示加载状态并阻止重复提交。生成完成后主操作变为“采用并保存到项目”或“继续修改”。

### 10.3 右侧任务队列与资产抽屉

进入项目创作台时，右侧只保留一个“项目资产”入口。点击后从右侧展开抽屉，不改变主画布宽度。抽屉同时承担两种职责：

1. 项目资产：母图、人物、道具、场景、分镜参考图和已保存成果；
2. 任务队列：待生成、生成中、已完成、失败、已取消。

每条任务包含缩略图、自然语言标题、提示词预览、状态、导入和操作。用户可以导入一条、发送一条，再导入下一条；队列提供取消和重试。批量模式提供“全部导入并生成”，但仍允许逐条确认。

## 11. 图片创作流程

### 11.1 项目进入

从项目节点进入图片创作台时，自动携带：

- 项目和课时；
- 目标资产位；
- 上一步生成的完整提示词；
- 参考图和风格约束；
- 默认比例和数量。

页面顶部显示归属，不再弹出“请选择项目”的重复选择框。

### 11.2 对话循环

`导入提示词 -> 发送 -> 生成候选 -> 放大/全屏查看 -> 上一张/下一张 -> 2 至 3 张候选对比 -> 采用 -> 保存到项目`

用户可以直接在下一条消息中修改“人物更小”“背景更干净”“保持上一版构图”等要求。每次生成保留可回看的要求和结果，但不把内部任务 ID暴露给教师。

## 12. 视频创作流程

### 12.1 视频不是另一套创作台

视频和图片使用同一底部输入区、结果对话和右侧任务队列。视频额外需要参考图、时长、镜头顺序和批量队列，不应重新设计成左右双栏表单。

### 12.2 项目视频链路

```text
母版剧本
  -> 粗分镜
  -> 画面风格
  -> 规划图片资产
      ├── 通用图片：母图 / 人物 / 道具 / 场景
      └── 分镜内部图：只属于具体镜头
  -> 图片创作台生成并保存资产
  -> 细分镜提示词
  -> 视频创作台生成片段
  -> 选择片段
  -> 合成完整视频
```

规划图片资产节点必须真实呈现母图、通用图片和分镜内部图的需求与提示词，不得用几张占位图代替设计信息。

### 12.3 视频任务队列

队列每条记录至少有：镜头标题、参考图、提示词、时长、状态、取消、重试、预览。用户可以将镜头提示词导入输入区，发送后留在当前创作台继续处理，不因导入而跳到另一个创作中心。

## 13. PPT 创作流程

PPT 流程是课时工作台中的独立分支，必须按顺序呈现：

```text
教案确认
  -> PPT 大纲
  -> 逐页设计稿
  -> 封面候选
  -> 正文页面
  -> 可编辑预览
  -> 导出 PPTX
```

封面确认后不能直接跳过大纲或设计稿。逐页设计稿展示每页的教学任务、页面结构、图片主视觉和关键数学信息；设计稿通过后才进入页面制作。正文页面为白色 16:9 画布，图片是主视觉，关键文字和数学图形可编辑。PPT 预览提供放大、全屏、上一页/下一页和缩略图导航，缩略图不承担文字检查。

## 14. 状态与用户语言

### 14.1 统一状态

| 内部状态 | 用户文案 | 语义色 | 可执行动作 |
| --- | --- | --- | --- |
| draft | 草稿 | 中性灰 | 编辑、继续 |
| waiting | 等待确认 | 琥珀 | 查看、确认 |
| running | 正在制作 | 蓝色 | 查看进度、取消 |
| succeeded | 已完成 | 绿色 | 预览、采用、保存 |
| failed | 制作失败 | 红色 | 查看原因、修改、重试 |
| stale | 内容已变化 | 紫灰或中性 | 查看影响、重新制作 |
| saved | 已保存到项目 | 绿色 | 查看资产、打开项目 |

状态必须同时有图标、文案和可访问文本。禁止只变颜色、只显示百分比或只显示技术 ID。

### 14.2 教师端词汇

| 内部概念 | 页面显示 |
| --- | --- |
| CreationPackage | 待制作内容 |
| CreationBatch | 本次创作 |
| Artifact | 作品 |
| Prompt | 完整生成指令 |
| NodeRun | 当前步骤 |
| review_required | 等待你确认 |
| stale | 内容已变化，建议更新 |
| shot_id | 镜头 1、镜头 2 |
| clip_id | 已采用片段 |

## 15. 视觉系统

### 15.1 风格关键词

高度简约、安静、专业、温和、有作品感。质感来自排版、比例、留白、细边界、层级和真实作品，而不是渐变、装饰性光晕或大面积卡片。

### 15.2 主题

主题由 `tokens.css` 统一控制，组件不得自行写颜色：

- 白色：纯白基底，适合白天和高对比工作。
- 护眼暖绿：低饱和暖绿色背景，适合长时间阅读教案。
- 暗灰：偏灰而非纯黑，适合夜间查看，正文对比度仍满足 WCAG AA。

主操作使用稳定的品牌绿色或中性深色，不使用廉价高饱和渐变。状态色固定为琥珀、蓝、绿、红、中性灰。

### 15.3 尺度

- 页面边距：桌面 24 至 32px，移动 16px。
- 普通控件高度：40px。
- 高密度工具栏：36px。
- 主要按钮上限：44px。
- 正文行高：1.6 左右；标题与正文保持清晰层级。
- 动效以 160 至 240ms 为主，支持 `prefers-reduced-motion`。

## 16. 交互与动效

- 路由节点切换使用内容层过渡，不卸载整个应用树。
- 左侧导航选中指示器使用平滑位移，保持滚动位置。
- 抽屉展开使用透明度和位移过渡，不使用大幅弹跳。
- 生成中显示稳定的骨架或进度状态，不能让页面高度突然变化。
- 结果生成后使用轻微淡入和尺寸稳定的占位，不能把作品推到首屏之外。
- 悬停只用于可点击、可拖拽或可提升的对象；普通卡片不统一上浮。
- 所有键盘操作有焦点状态，所有图标按钮有 Tooltip 和可访问名称。

## 17. API 与数据边界

前端只通过生成的 OpenAPI 类型和 feature API 客户端访问后端：

- 项目：`projectsApi`；
- 教材：`materialsApi`；
- 课时：`lessonsApi`；
- 工作流：`workflowApi`；
- 作品与版本：`artifactsApi`；
- 项目资产位：`assetsApi`；
- 创作批次与结果：`creationApi`；
- 任务与事件流：`jobsApi`、项目 SSE 和 Job SSE。

写操作使用 `Idempotency-Key`；编辑冲突使用 ETag 与 `If-Match`；SSE 断线使用 `Last-Event-ID` 恢复。浏览器不保存 Provider 密钥，不在 UI 展示供应商任务 ID。Mock 仅用于开发和验收，Runtime 未接通的页面必须明确不可用，不得伪装成真实生产闭环。

## 18. 组件实现约束

统一使用：

- `shared/ui/Button.tsx`：所有文字按钮；
- `shared/ui/IconButton.tsx`：图标操作；
- `shared/ui/Select.tsx`：所有下拉选择；
- `StatusBadge`：状态语义；
- `FocusPageHeader`：页面标题和唯一主操作；
- `HorizontalScrollArea`：移动端横向导航和候选列表。

禁止在页面内另写同功能按钮、选择框、状态徽章或颜色值。新增组件先判断是否能扩展现有注册表：节点使用 `nodeRendererRegistry`，字段使用 `fieldRegistry`，预览使用 `artifactPreviewRegistry`，创作能力使用 `studioRegistry`。

## 19. 验收标准

### 19.1 视觉验收

- 320、375、414、768、1280 像素宽度无横向溢出。
- 100% 浏览器缩放下首页、项目页、工作台和创作台的主作品与主操作可见。
- 不出现大面积空白、作品缩成缩略图、固定栏遮挡正文或卡片套卡片。
- 图片、PPT 和视频均能在不下载的情况下检查主要细节。

### 19.2 交互验收

- 相邻流程节点切换不整页刷新、不回到顶部、不丢失编辑内容。
- 创作台可以输入、发送、重新生成、取消、重试、采用和保存。
- 项目进入创作台自动挂载项目、课时、资产位和提示词。
- 右侧资产抽屉不挤压主画布，队列状态实时或可恢复更新。
- PPT 封面之后必须经过大纲和逐页设计稿。
- 每个页面只有一个明确主操作，返回上一步在存在上游节点时可用。

### 19.3 质量验收

- `npm run format`
- `npm run lint`
- `npm run typecheck`
- `npm test -- --maxWorkers=2`
- `npm run build`
- Playwright 覆盖首页、项目列表、工作台、图片创作、视频创作和 PPT 流程。
- axe 检查无嵌套 main、重复 landmark、标题层级跳跃和无名称图标按钮。

## 20. 优化执行顺序

1. 统一最新分支、路由和运行入口，避免同时优化两套前端。
2. 先修工作台导航无刷新、滚动和节点聚焦。
3. 再完成通用创作台的对话式展示、底部输入和资产任务抽屉。
4. 补齐 PPT 逐页设计稿和视频规划图片资产节点。
5. 重构首页和项目列表的信息密度。
6. 最后收紧主题、圆角、按钮、状态、动效和响应式细节。
7. 以截图、键盘操作和真实用户路径做最终验收，不以“页面能打开”作为完成标准。
