# 文档与仓库内容生命周期

本文件解决三个问题：信息应该放在哪里、什么时候更新、什么时候删除。目标是让当前树只包含未来开发仍需要读取的现行事实。

## 1. 文件职责

| 位置 | 唯一职责 | 不允许包含 |
| --- | --- | --- |
| `README.md` | 稳定项目介绍、技术基线和入口 | 当前任务、开发日志、版本历史 |
| `AGENTS.md` | 全仓强制开发总则 | 产品细节、任务流水 |
| `CURRENT_STATUS.md` | 当前里程碑、活跃工作、阻塞和下一出口 | 历史事件、完整Issue内容 |
| `docs/START_HERE.md` | 按角色提供最短阅读路径 | 第二套需求说明 |
| `docs/product/` | 当前产品范围和行为 | 实施日志、旧版本附录 |
| `docs/workflows/` | 当前业务流程和产物关系 | Provider实现细节 |
| `docs/frontend/` | 当前前端体验与工程约束 | 后端内部实现 |
| `docs/backend/` | 当前后端、数据和运行约束 | 临时任务计划 |
| `docs/governance/` | 团队协作、文档政策和路线 | 每日状态和项目历史 |
| `docs/governance/协作机制复盘与防复发.md` | 已证实的协作失败机制、现行替代动作和验证状态 | 完整会话、逐任务流水、一次性SHA清单 |
| `contracts/` | OpenAPI、JSON Schema和共享约定 | 仅供人阅读的模糊要求 |
| `.github/` | Issue、PR、CODEOWNERS和CI入口 | 产品规格副本 |
| `deliverables/` | 当前明确需要对外发送的包 | 按日期累计的归档 |

任务、讨论、候选方案、交接和短期计划进入GitHub Issue/PR，不进入`main`文档。

## 2. 新文档准入

新增文档前必须回答：

1. 哪类读者会在未来任务中反复读取？
2. 为什么现有文档不能承载？
3. 它是哪一类事实的唯一来源？
4. 谁在什么变更发生时更新？
5. 被替代时如何删除？

无法回答时不新增文件，改用Issue或PR。

一个文档只承担一个责任。超过300行时必须按责任拆分，或者在PR中说明保持单文件的原因。

## 3. 命名规则

正式文档使用稳定的语义名称。禁止：

```text
V1 / V2 / V3
final / final-final / latest / new
日期副本
补充说明
旧版备份
copy / backup / 临时
```

版本信息保存在Git、Manifest、内容发布记录或Schema字段中，不进入目录名和文件名。

## 4. 修改现行口径

小型澄清直接修改唯一现行文件。

改变合同、数据或产品主线时：

1. 创建Decision Issue。
2. 确认影响和迁移方式。
3. 在一个PR内修改所有受影响的现行文档和合同。
4. 同步实现、迁移和测试，或在`CURRENT_STATUS.md`明确尚未实现的阻塞。
5. 删除被替代的文件、字段和入口。

不得保留两套冲突口径让开发者自行判断。

## 5. 历史与决策

- Git历史保存文件旧版本。
- Issue保存问题、候选方案和讨论。
- PR保存实施差异、审核和测试证据。
- 当前文档只解释现在为什么这样工作，不追加逐版本变更记录。

新对话默认不读取历史。只有追查回归、审计责任或理解当前设计无法解释的决定时，才按Issue、PR或`git blame`定向查询。

协作复盘只保留仍会影响未来任务的失败机制、根因、替代动作、状态和证据入口。单次命令输出、完整时间线和已经失去维护价值的事件留在Issue/PR或Git历史；同类机制变化时原位修改唯一复盘，不新增日期或版本副本。

## 6. 临时材料

以下内容不进入正式目录：

- 探索性草稿。
- 一次性实施清单。
- 模型原始响应。
- 临时提示词实验。
- 会议流水和聊天摘要。
- 本地调试日志。
- 截图、视频和媒体中间产物。
- 用户上传教材和真实业务数据。

临时材料使用操作系统临时目录、受控对象存储或Issue/PR。需要转为长期事实时，提炼后更新现行文档，不复制整个讨论过程。

## 7. 代码和生成文件

- 源码按业务模块或Feature组织。
- 生成文件必须集中到明确目录，并能通过命令重新生成。
- 不提交本地缓存、依赖目录、运行日志、数据库和媒体输出。
- 不保留注释掉的旧实现或手工备份。
- 小型、脱敏、拥有明确测试用途的Fixture可以提交。

临时生成内容进入系统临时目录；正式项目资产进入对象存储；当前对外交付包进入`deliverables/`。

## 8. 对外交付包

- `deliverables/`只保留当前明确需要的包。
- 新包替换旧包，不以日期或版本号累计副本。
- 包内必须有入口、范围、校验和与生成日期。
- 包不是事实源；相同内容仍以仓库现行文件为准。
- 外包或第三方返回源码后，必须通过PR进入正式代码目录，不能以ZIP长期代替源码。

## 9. CURRENT_STATUS维护

状态页只在以下事件更新：

- 进入新里程碑。
- 可演示能力发生实质变化。
- 新增或解除阶段阻塞。
- 下一阶段出口改变。

普通提交、每日进度和个人工作不更新状态页。状态页引用Issue/PR，不复制正文。

状态页包含需要产品和工程判断的语义，禁止由机器人根据提交数量或Issue标签直接覆写。自动化只负责两类可证明事实：拒绝状态页与当前树中的已实现基座矛盾，以及校验PR是否明确声明并履行上述更新责任。语义校准仍由任务负责人完成，并由`CURRENT_STATUS.md`的CODEOWNER审核。

## 10. 删除与清理

满足任一条件时，在造成该变化的同一PR删除文件：

- 内容已由另一个现行来源完整承担。
- 文件描述已失效设计。
- 文件只是完成任务的临时计划。
- 外部交付包已被当前包替换。
- 生成物可以稳定重建且不属于发布物。

删除不等于丢失，Git历史仍可定向恢复。禁止为了“可能以后有用”把失效内容留在当前入口。

## 11. 文档评审清单

- 是否修改了正确的唯一来源？
- 是否产生重复或相反口径？
- 是否删除了失效入口？
- 是否把任务过程误写进长期文档？
- 是否存在版本、日期、final或copy命名？
- 链接、Schema、示例和代码是否同步？
- 新成员是否能只读现行内容完成任务？
