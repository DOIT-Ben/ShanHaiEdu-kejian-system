# 分支、合同与变更规则

## 分支

- `main` 是唯一长期活动分支，必须保持可构建、可测试。
- 功能开发使用 `feature/<scope>`，修复使用 `fix/<scope>`，迁移使用 `migration/<scope>`。
- 承接方通过 Pull Request 合并；不得向 `main` 强推或重写历史。
- 合并方式建议 squash merge，提交信息说明业务结果。

## 事实来源

- 当前产品与运行时设计以`docs/superpowers/specs/2026-07-17-shanhai-courseware-platform-runtime-design.md`及后续已批准ADR为准。
- 后端提交的 OpenAPI 是联调唯一接口事实来源。
- 数据库变化必须带 Alembic migration。
- 工作流变化必须升级 definition 版本，运行中的项目继续绑定旧版本。
- Prompt 模板变化必须形成新版本，不覆盖历史运行使用的内容。
- 内容结构、规则、锚点和模型档案变化必须形成新的内容包版本，项目继续固定引用原版本，除非显式迁移。
- Figma 只在标记 Ready for Dev 后成为视觉验收依据。

## 兼容规则

- 删除字段、改名、修改枚举语义和改变状态转换均属于破坏性变更。
- 破坏性变更必须给出影响、迁移、过渡期和回滚方案。
- 前端不得猜测缺失字段；后端不得静默改变含义。
- API 错误必须使用稳定错误码，文案可以优化但错误码语义不能漂移。

## 密钥与数据

- 禁止提交 `.env`、密钥、Token、私有 URL、教师教材、学生个人信息和模型敏感原始响应。
- 示例配置只使用明显的占位值。
- 日志和测试证据必须脱敏。
- 大型生成产物不进入 Git；正式产物进入对象存储，版本化交付包可进入 `deliverables/`。

## Pull Request 最低要求

- 变更目的和范围。
- 影响的页面、API、数据表、工作流或 Provider。
- 已运行的验证命令和结果。
- UI 变更提供截图；API 变更提供 OpenAPI diff。
- 风险、回滚方式和未完成事项。
- 不得把 Mock、占位文件或未运行测试描述为正式完成。
