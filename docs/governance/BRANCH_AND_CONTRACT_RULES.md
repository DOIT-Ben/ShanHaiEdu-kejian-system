# 分支、合同与版本治理

## 1. 仓库口径

`main` 始终只保留一套现行设计和一套现行合同。正式目录使用稳定名称，不创建 `v1`、`v2`、`final`、`最新版` 或日期副本。历史由 Git 提供，当前树不通过并行文档保存历史。

设计优先级：

1. `docs/product/PRODUCT_SPEC.md` 与 `docs/workflows/` 中的业务语义；
2. `contracts/` 中的跨前后端机器合同；
3. `docs/backend/`、`docs/frontend/` 专项设计；
4. 实现代码、迁移和生成文档必须符合以上口径。

发现冲突时停止实现并统一上层定义，不得在代码中兼容两套相反口径。

## 2. 分支策略

- `main`：可部署、合同自洽、测试通过。
- 功能分支：`feat/<scope>-<short-name>`。
- 修复分支：`fix/<scope>-<short-name>`。
- 文档重构也走短分支并整体替换；用户明确要求直接主分支交付时，可在完整验证后推送主分支。
- 禁止把供应商构建产物、密钥、真实教材、模型原始敏感响应提交仓库。

提交应按可审核的业务切片组织。破坏性数据库或合同变更必须在同一变更中包含迁移、消费者更新、测试和回退方案。

## 3. 合同治理

- REST 以 `/api/v2` 为当前基线。
- OpenAPI 由后端实现生成，但必须满足 `contracts/` 的稳定约定和 Schema。
- 前端只从当前 OpenAPI 生成类型，不维护手写重复 DTO。
- SSE 事件、错误包、节点状态、创作包和动态内容定义使用根目录 JSON Schema。
- 枚举新增通常是兼容扩展，但消费者必须有未知值兜底；删除、重命名是破坏性变更。
- 字段从可选变必填、改变语义或收窄取值均视为破坏性变更。

CI 必须做 OpenAPI diff、JSON Schema 校验和 Mock 场景一致性检查。

## 4. 内容包与业务版本

内容包和工作流发布后不可变。管理员修订时发布新版本；项目固定 `content_release_id` 和 `workflow_definition_version_id`。现有项目只通过显式迁移改变版本，迁移前展示字段、节点、资产槽位和提示词差异。

禁止在代码发布中偷偷修改线上项目使用的提示词或教案结构。紧急安全规则可以平台级覆盖，但必须审计、说明影响并随后形成正式内容发布。

## 5. 数据库迁移

- 只使用 Alembic，禁止生产自动建表。
- 采用扩展—回填—切换—清理，至少跨一个兼容发布后再删除旧结构。
- 迁移不得依赖不确定的模型输出。
- 大表迁移必须估算锁和时长，必要时分批回填。
- 每次发布在备份可恢复验证后执行迁移。

## 6. 外包协作

外包前端只读 `docs/frontend/`、`contracts/` 和 `docs/product/PRODUCT_SPEC.md`。任何口头变更必须先进入上述现行文件再开发。供应商不得用新的“补充说明”覆盖正式文档，也不得复制一份合同私下修改。

交付必须包含源代码、测试、Storybook、生成类型、运行说明和验收清单；截图和静态 HTML 只能作为说明，不算功能交付。
