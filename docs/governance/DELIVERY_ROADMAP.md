# 渐进式开发与交付路线

目标不是先完成所有基础设施，也不是同时铺开所有模块，而是用最小公共基座支撑一条条可演示的真实用户链路。根据[Decision #199](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/199)与[Decision #204](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/204)，阶段编号是能力地图，不是自动开工队列；当前先完成教案页面与真实短片黄金节点组成的教师可见MVP，再继续生产加固。

## 1. 开发原则

- 先交付能改变教师行为的纵向业务链，再补只被真实使用证明必要的共享能力。
- 按步骤辅助创作运行时先于完全无人值守的`automatic`策略。
- 一键推进到下一个审核点先于无人值守全流程。
- 前端可以基于合同和MSW整体推进；后端按可验收链路逐步集成。
- 当前先完成教材到教案与三类九套的教师页面，再用一个真实短片黄金节点证明视频生成、播放、采用和项目回写；PPT、完整图片链、完整视频链和TTS均在首个MVP后由独立Issue重新定界。
- 普通CI使用确定性Fake；阶段验收使用真实模型和真实文件。
- 不以Mock、截图或静态原型宣称里程碑完成。

### 当前唯一执行顺序

1. 合并#4/PR #111的生产前端源码，只保留首条MVP链需要的页面和体验。
2. 由#11完成唯一真实页面：选择已上传教材和物理页码，经服务端调用真实`newapi/deepseek`，展示课时划分、每课时十二部分教案和三类九套导入方案，支持编辑、局部重生成、批准和唯一选择。
3. 由#205在一个跨栈短分支和PR内新增`1.5.0`最小前向绑定，并在现有教师页面完成“上传或选择唯一关键帧 -> 生成6秒级真实短片 -> 轮询状态 -> 播放 -> 采用 -> 刷新恢复”。#206并入#205，不保留“后端完成、前端后接”的中间出口。
4. 用一个受控真实小学数学项目验证正式页面、正式API、PostgreSQL、真实文本/视频Provider、真实文件和真实保存，不能由Mock、fixture、CLI或静态替身代替。
5. 由未参与实现的教学验收子智能体审查，并根据意见优化一轮后复验完整链路。
6. 首个MVP关闭后，再分别评估PPT/PPTX、完整图片链、完整视频链、TTS、自动重试和SSE hardening。

最低可靠性只保留租户与权限边界、PostgreSQL业务真源、服务端Provider调用、正式文件事实、教师采用、刷新恢复、密钥脱敏和单次生成预算；已有幂等、事务与运行时能力直接复用，不再扩建。除承载黄金视频节点所必需的`1.5.0`最小前向绑定外，新的发布版本链、极端并发、Worker恢复、自动重试、SSE hardening、PPT/PPTX、完整媒体链、生产扩容和全量可观测性延后，除非真实页面或黄金短片出现可复现阻断。

当前资源固定为70%页面与教师体验、20%正式API联调、10%必要安全与回归。连续投入后台底层却未让该页面更接近可用时，必须暂停并转入后续Hardening Issue。

### 固定责任分工

- Codex负责除生产前端之外的平台：`apps/api`、`workers`、`workflow`、`infra`、数据库、对象存储、模型网关、共享合同、生成客户端命令和后端CI。
- 独立前端承接方只负责`apps/web`、Figma、Storybook、MSW、前端测试、前端部署和真实API联调。
- Codex不得为了演示在`apps/web`堆临时页面；外包方不得实现服务端业务、数据库、任务队列或模型调用。
- 双方只通过OpenAPI、JSON Schema、SSE、错误合同、Mock场景和关联Issue协作。
- 前端发现合同缺口时提交复现、影响和建议，由Codex负责的合同任务统一修改；前端不得建立私有DTO或改变业务语义。

## 2. 阶段0：协作与工程基线

### 目标

让团队可以在同一口径下安全开发、审核、交接和清理。

### 范围

- README、AGENTS、CURRENT_STATUS和治理文档。
- Issue/PR模板、CODEOWNERS、标签和Milestone规则。
- `main`分支保护和合并后自动删除分支。
- Monorepo目标目录和基础开发命令。
- 前后端基础CI、合同校验和仓库清洁检查。

### 可见成果

新成员可以从Issue领取任务，启动开发环境，提交一个满足检查的PR，并由另一位成员按标准接手。

### 出口

- 治理规则已合并并经过一次真实任务验证。
- 前后端骨架、本地依赖和CI能够启动。
- OpenAPI、JSON Schema、格式、类型和基础测试进入自动门禁。

## 3. 阶段1：最小可运行基座

### 目标

打通浏览器、API、数据库、任务队列、对象存储和模型网关。

### 范围

- React应用接入真实API和MSW模式。
- FastAPI模块化单体、PostgreSQL、Redis、对象存储和Worker。
- 用户、项目、课时、资产、版本和任务基础模型。
- 文件上传、任务状态、SSE进度、取消和错误反馈。
- Prompt快照、模型调用审计和一个真实文本模型适配器。

### 可见成果

教师可以登录、创建项目、上传测试教材，并实时看到解析任务状态和项目资产记录。

### 出口

- 一条上传任务从前端到Worker再回到SSE完整运行。
- 失败、重试、取消和幂等至少有集成测试。
- 密钥只存在服务端受控环境。
- 数据库可通过Alembic从空库升级。

## 4. 阶段2：教材到教案纵向链路

### 目标

交付第一个真实教师价值闭环。

### 链路

```text
选择已上传单知识点教材与物理页码
→ PDF解析
→ 课时划分
→ 审核修改
→ 分课时教案生成
→ 动态结构编辑
→ 批准并版本化
→ 三类九套导入方案生成
→ 编辑、局部重生成与批准
→ 完成每课时唯一选择
→ 保存项目资产
```

### 要求

- 使用真实文本模型。
- 提示词可见、可改并保存快照。
- 默认12部分由内容定义驱动，不能写死在页面或数据库列中。
- 三类九套导入作为独立版本化附属成果。
- 每个节点支持返修、重新生成和保留历史版本。
- 内容包运行时先支持内置受控包；管理端导入界面可以后置。
- 首套内置包当前正式发布为`1.0.0`；#130与PR #141建立47节点v2候选，#146完成安全与发布固定性加固，#116再把Intro批准与选择拆为独立节点，使`1.1.0`候选成为48节点目录（22个`model_generation`、13个`deterministic`、13个`human_gate`）。该候选只有显式PostgreSQL发布后才成为后续新项目默认；黄金Fixture只作为确定性开发基线，阶段出口仍使用真实教材和真实文本模型。

### 当前实施边界

教材、教案和Intro的公共运行时前置已经进入`main`。#118已关闭真实页面不能绕过的`material_scope`正式输出与教师保存缺口；这些能力现在直接服务于浏览器MVP联调，不再作为继续扩张后台合同的理由。普通CI始终使用确定性Fake，MVP出口使用受控真实Provider和真实文件，两类证据不可互相替代。

### 可见成果

教师从教材得到可编辑、可批准、可追溯的多课时教案和导入方案。

### 出口

- 一个黄金教材样例完成整条链路。
- 结构化JSON、教师可读投影和资产版本一致。
- 教师批准和唯一选择写入正式事实，刷新页面后结果仍然存在。

## 5. 阶段3：PPT与完整媒体后续链路

PPT、PPTX、完整图片链、完整视频链和TTS均不属于首个教师MVP。首个MVP只包含Decision #204批准的一个真实短片黄金节点；不得把既有deterministic运行时、CLI smoke或媒体Fixture描述为教师可见产品出口。

### PPT工作流

```text
批准教案
→ PPT大纲
→ 逐页设计
→ 封面
→ 视觉规范
→ 正文图片
→ 页面预览与修改
→ 可编辑PPTX
```

验收重点：封面独立设计，正文纯白，每页有图像主视觉，关键文字、公式和标注保持可编辑。

### 视频工作流

首个MVP只实现以下稳定切片：

```text
exact已选导入方案 + 一个正式关键帧
→ video.shots.generate真实Provider
→ MP4文件事实校验
→ 页面轮询并播放候选
→ 教师采用
→ 项目课堂导入视频槽位
```

当前不可变`1.4.0`中的`video.shots.generate`要求`artifact:video_fine_storyboard`、`contract:video_style`并依赖`video.fine_storyboard.generate`，不能直接用于该切片。#205必须生成`1.5.0`前向候选：该节点只冻结exact `intro_selection.snapshot`与教师上传或选择的一个租户内、clean、正式`shot_keyframe`图片版本，成为视频分支入口；不要求先建设图片生成或Artifact批准链。风格使用发布内固定预设，候选数固定为1，输出继续沿用现有视频候选、采用和项目槽位合同。旧`1.0.0`至`1.4.0`及既有项目绑定不变，只有新建或显式升级的MVP项目可绑定`1.5.0`。

后续完整链在同一NodeRun、GenerationResult/Adoption、FileAssetVersion和项目槽位上扩展：

```text
选定导入方案快照
→ 母版剧本
→ 粗分镜
→ 视觉母版
→ 四类图片资产
→ 细分镜与提示词
→ 候选视频片段
→ 保存正式片段
→ 音频字幕
→ MP4合成
```

验收重点：图片先于细分镜，候选结果被选择并保存后才产生正式`clip_id`，视频不得读取完整教案或教材。

黄金节点不以#175的完整成片声明、#176的逐镜头血缘或#189的90秒全节点验收作为前置。后续只替换或扩展黄金节点的上游输入生产与多镜头编排，不推翻已经验证的文件、候选、采用、项目槽位和页面播放合同。真实短片必须来自正式Provider；fixture、历史MP4或静态视频不能替代。

### 共享能力

- 模型网关和Provider适配。
- Prompt编译和快照。
- 任务、预算、重试和SSE。
- 项目资产、版本、创作任务包和原子保存。
- 图片、视频与PPT通用创作中心。
- 审核、质量规则和真实产物验证。

PPT和视频不得复制共享能力或维护不同状态机。

## 6. 阶段4：教师可见小学数学MVP

首个正式MVP必须在一个生产教师页面完成一次真实项目：

```text
选择已上传教材和物理页码
→ 服务端调用真实newapi/deepseek
→ 展示课时划分
→ 展示每课时十二部分教案
→ 展示三类九套导入方案
→ 编辑或局部重生成
→ 批准教案与导入方案
→ 完成唯一选择
→ 上传或选择一个正式关键帧
→ 生成并轮询一个真实短片
→ 播放并采用到项目
→ 刷新后恢复正式结果
```

### MVP边界

- 只闭环小学数学和一个小知识点项目。
- 文本使用一个稳定真实Provider，浏览器不得直连Provider。
- 先交付一套验证过的内容包。
- 先交付按步骤辅助创作和“一键推进到审核点”，两者引用同一工作流定义。
- 用户可以编辑公开业务内容、局部重生成、批准并完成唯一选择。
- 视频首版只有一个短片黄金节点，使用正式Provider和真实MP4；不要求多镜头、TTS、字幕或最终合成。
- 正式API、PostgreSQL和刷新恢复共同证明保存成立。
- Mock只用于开发测试，不是验收结果。
- PPT/PPTX、完整图片链、完整视频链、TTS、失败重试和SSE hardening全部后置。

### MVP出口

- 新教师无需离开平台完成黄金项目。
- 教师在浏览器看到的教材页码、课时划分、十二部分教案、三类九套、批准、唯一选择和当前课堂导入视频均来自正式API与持久化事实。
- 真实`newapi/deepseek`、真实视频Provider、可播放MP4、PostgreSQL、采用回写和刷新恢复全部通过。
- 未参与实现的教学验收子智能体完成审查，并按意见优化一轮后复验完整链路。
- 脱敏记录基本成本、耗时和Provider请求，不要求首版完成全量运营观测。

## 7. 阶段5：自动化与产品化

MVP通过后增加：

- PPT/PPTX、完整图片链、完整视频链和TTS纵向链路。
- 真实失败重试、SSE hardening与更完整的恢复矩阵。
- 完全自动审批策略。
- 多模型路由、限流和故障切换。
- 预算、质量与风险自动暂停。
- 内容包、提示词和工作流的管理端导入发布。
- 批量项目与批量生成。
- 完整权限、审计和运营能力。
- 性能、并发、缓存和成本优化。
- 更多学科、教材和内容包。
- 完整音频权利声明、逐镜头lineage/stale和90秒视频全节点验收。

完全自动不是第二套工作流，只是同一运行时上的审批与推进策略。

## 8. 并行策略

### 受控阶段重叠

当一个阶段只被独立的前端或消费者联调门禁阻塞时，可以通过Decision Issue授权下一阶段中不依赖该门禁的后端基础任务提前启动。受控重叠必须同时满足：

- 前一阶段和被阻塞任务继续保持未完成状态，不得用后端测试替代浏览器或消费者验收。
- 下一阶段使用独立Milestone、父Issue和可独立验收的子Issue；每个子Issue仍使用短分支、仓库外worktree和Draft PR。
- 合同、核心数据和统一状态机先于业务分支；依赖未满足的任务保持`status:ready`。
- 所有阶段合计同时进行的后端主任务不超过三个；普通CI使用确定性Fake，Provider变更和阶段出口使用真实冒烟。
- 发现产品、合同或数据语义冲突时停止受影响任务，回到Decision Issue裁决，不同时支持相反口径。

Decision #18曾授权在前端阻塞时提前推进后端基座；[Decision #199](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/199)与[Decision #201](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/201)结束该默认扩张期。[Decision #204](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/204)只授权一个真实短片黄金节点，不恢复后续底座或完整媒体链。

### 可以并行

- 前端基于已发布OpenAPI、Schema和MSW实现完整界面。
- PR #111是`apps/web`唯一活动现场；#11只在其合并后建立唯一正式API联调现场。
- #207合并后，#205可以立即建立唯一跨栈分支和Draft PR，并行实现不触碰`apps/web`的`1.5.0`最小前向绑定、视频runtime/API与后端测试；PR #111和#11仍独占`apps/web`，释放后#205才可在同一分支接入教师页面。#205在页面闭环完成前不得转Ready或合并，也不得把后端阶段描述为MVP完成。
- 普通测试、MVP浏览器验收准备和独立只读审查可以跟随所属任务推进。
- 自动化测试和最低安全检查跟随所属业务任务同步开发；可观测性扩张必须由真实MVP缺口驱动。

### 不能并行绕过

- 数据所有权未确定前，不分别设计多套表结构。
- 状态机未统一前，不分别实现PPT和视频任务状态。
- 项目资产合同未稳定前，不分别实现创作中心回写。
- Prompt、Schema和内容包没有版本规则前，不把模板写死进业务代码。

同时进行的后端主工作流不超过三个。并行团队通过合同和子Issue协作，不通过长期前端、后端或个人分支协作。

## 9. 阶段验收与调整

每个阶段结束必须具备：

- 可运行软件。
- 可演示用户结果。
- 自动化测试证据。
- 可执行数据迁移。
- 现行文档、合同和实现一致。
- 已知风险、成本和阻塞清单。
- 更新后的`CURRENT_STATUS.md`和下一阶段Issue集合。

路线允许根据真实实现调整。调整通过Decision Issue进入，确认后直接修改本文件和受影响的现行合同，不追加版本历史。
