# 渐进式开发与交付路线

目标不是先完成所有基础设施，也不是同时铺开所有模块，而是用最小公共基座支撑一条条可演示的真实用户链路。根据[Decision #199](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/199)，阶段编号是能力地图，不是自动开工队列；当前先完成教师可见MVP纵向闭环，再继续生产加固。

## 1. 开发原则

- 先交付能改变教师行为的纵向业务链，再补只被真实使用证明必要的共享能力。
- 按步骤辅助创作运行时先于完全无人值守的`automatic`策略。
- 一键推进到下一个审核点先于无人值守全流程。
- 前端可以基于合同和MSW整体推进；后端按可验收链路逐步集成。
- 公共合同稳定后，PPT和视频可以并行。
- 普通CI使用确定性Fake；阶段验收使用真实模型和真实文件。
- 不以Mock、截图或静态原型宣称里程碑完成。

### 当前唯一执行顺序

1. 收口已经在做且直接贡献真实PPTX出口的#170，不新增底座范围。
2. 合并#4/PR #111的生产前端源码并完成#11真实API联调。
3. 通过#107/#108让教师从项目生成、预览并下载真实可编辑PPTX。
4. 用一个受控真实小学数学项目完成浏览器纵向验收：创建项目、上传教材、查看真实进度、打开并修改/批准教案与PPT、下载PPTX、重试一次失败任务并刷新恢复。
5. 首个MVP通过后，再以一个短课堂导入片段验证最小视频链；#175、#176和#189原有生产级范围不自动恢复。

最低可靠性包括租户与权限边界、PostgreSQL业务真源、正式文件事实、任务幂等、失败可见、一次重试、SSE/刷新恢复、服务端Provider调用、基本预算上限和PPTX下载资格。新的发布版本链、镜头级血缘、完整音频权利声明、90秒全节点视频、生产扩容和全量可观测性延后，除非真实MVP反馈证明其为阻断项。

### 固定责任分工

- Codex负责除生产前端之外的平台：`apps/api`、`workers`、`workflow`、`infra`、数据库、对象存储、模型网关、共享合同、生成客户端命令和后端CI。
- 独立前端承接方只负责`apps/web`、Figma、Storybook、MSW、前端测试、前端部署和真实API联调。
- Codex不得为了演示在`apps/web`堆临时页面；外包方不得实现服务端业务、数据库、任务队列或模型调用。
- 双方只通过OpenAPI、JSON Schema、SSE、错误合同、Mock场景和关联Issue协作。
- 前端发现合同缺口时提交复现、影响和建议，由Codex负责的合同任务统一修改；前端不得建立私有DTO或改变业务语义。

## 2. 阶段0：协作与工程基线

### 目标

让团队可以在同一口径下安全开发、审核、交接和清理。

### 范围

- README、AGENTS、CURRENT_STATUS和治理文档。
- Issue/PR模板、CODEOWNERS、标签和Milestone规则。
- `main`分支保护和合并后自动删除分支。
- Monorepo目标目录和基础开发命令。
- 前后端基础CI、合同校验和仓库清洁检查。

### 可见成果

新成员可以从Issue领取任务，启动开发环境，提交一个满足检查的PR，并由另一位成员按标准接手。

### 出口

- 治理规则已合并并经过一次真实任务验证。
- 前后端骨架、本地依赖和CI能够启动。
- OpenAPI、JSON Schema、格式、类型和基础测试进入自动门禁。

## 3. 阶段1：最小可运行基座

### 目标

打通浏览器、API、数据库、任务队列、对象存储和模型网关。

### 范围

- React应用接入真实API和MSW模式。
- FastAPI模块化单体、PostgreSQL、Redis、对象存储和Worker。
- 用户、项目、课时、资产、版本和任务基础模型。
- 文件上传、任务状态、SSE进度、取消和错误反馈。
- Prompt快照、模型调用审计和一个真实文本模型适配器。

### 可见成果

教师可以登录、创建项目、上传测试教材，并实时看到解析任务状态和项目资产记录。

### 出口

- 一条上传任务从前端到Worker再回到SSE完整运行。
- 失败、重试、取消和幂等至少有集成测试。
- 密钥只存在服务端受控环境。
- 数据库可通过Alembic从空库升级。

## 4. 阶段2：教材到教案纵向链路

### 目标

交付第一个真实教师价值闭环。

### 链路

```text
上传单知识点教材
→ PDF解析
→ 课时划分
→ 审核修改
→ 分课时教案生成
→ 动态结构编辑
→ 批准并版本化
→ 保存项目资产
```

### 要求

- 使用真实文本模型。
- 提示词可见、可改并保存快照。
- 默认12部分由内容定义驱动，不能写死在页面或数据库列中。
- 三类九套导入作为独立版本化附属成果。
- 每个节点支持返修、重新生成和保留历史版本。
- 内容包运行时先支持内置受控包；管理端导入界面可以后置。
- 首套内置包当前正式发布为`1.0.0`；#130与PR #141建立47节点v2候选，#146完成安全与发布固定性加固，#116再把Intro批准与选择拆为独立节点，使`1.1.0`候选成为48节点目录（22个`model_generation`、13个`deterministic`、13个`human_gate`）。该候选只有显式PostgreSQL发布后才成为后续新项目默认；黄金Fixture只作为确定性开发基线，阶段出口仍使用真实教材和真实文本模型。

### 当前实施边界

教材、教案和Intro的公共运行时前置已经进入`main`。这些能力现在直接服务于浏览器MVP联调，不再作为继续扩张后台合同的理由。#118只保留受控真实教材、真实文本生成、预算和脱敏证据中能验证首条用户链的部分；普通CI始终使用确定性Fake，MVP出口使用受控真实Provider和真实文件，两类证据不可互相替代。

### 可见成果

教师从教材得到可编辑、可批准、可追溯的多课时教案和导入方案。

### 出口

- 一个黄金教材样例完成整条链路。
- 结构化JSON、教师可读投影和资产版本一致。
- 项目刷新或任务重试不会产生重复正式资产。

## 5. 阶段3：PPT优先与最小视频验证

当前先完成PPT，因为真实可编辑PPTX是首个教师MVP的核心交付物。视频不再与PPT自动并行；首个MVP通过后，先验证一个短课堂导入片段的生成、播放、采用和项目回写，再决定是否需要恢复完整生产链。

### PPT工作流

```text
批准教案
→ PPT大纲
→ 逐页设计
→ 封面
→ 视觉规范
→ 正文图片
→ 页面预览与修改
→ 可编辑PPTX
```

验收重点：封面独立设计，正文纯白，每页有图像主视觉，关键文字、公式和标注保持可编辑。

### 视频工作流

```text
选定导入方案快照
→ 母版剧本
→ 粗分镜
→ 视觉母版
→ 四类图片资产
→ 细分镜与提示词
→ 候选视频片段
→ 保存正式片段
→ 音频字幕
→ MP4合成
```

验收重点：图片先于细分镜，候选结果被选择并保存后才产生正式`clip_id`，视频不得读取完整教案或教材。

首轮视频只取上述链路的最小可验证切片，不以#175的完整成片声明、#176的逐镜头血缘或#189的90秒全节点验收作为前置。真实短片必须来自正式Provider并可播放、采用和回写；fixture或历史MP4不能替代。

### 共享能力

- 模型网关和Provider适配。
- Prompt编译和快照。
- 任务、预算、重试和SSE。
- 项目资产、版本、创作任务包和原子保存。
- 图片、视频与PPT通用创作中心。
- 审核、质量规则和真实产物验证。

PPT和视频不得复制共享能力或维护不同状态机。

## 6. 阶段4：教师可见小学数学MVP

首个正式MVP必须完成一次真实项目：

```text
上传教材
→ 自动划分课时
→ 生成并批准教案
→ 生成完整PPT
→ 浏览器预览、修改或批准
→ 导出真实可编辑PPTX
→ 制造一次失败并完成重试与刷新恢复
```

### MVP边界

- 只闭环小学数学和一个小知识点项目。
- 文本和PPT背景各使用一个稳定Provider。
- 先交付一套验证过的内容包。
- 先交付按步骤辅助创作和“一键推进到审核点”，两者引用同一工作流定义。
- 用户可以编辑公开业务内容并批准结果。
- PPTX真实可打开、可编辑和可下载。
- Mock只用于开发测试，不是验收结果。
- 课堂导入视频是首个MVP后的第二条纵向切片，不阻塞PPT MVP验收。

### MVP出口

- 新教师无需离开平台完成黄金项目。
- 教师在浏览器看到的进度、错误、教案、PPT和下载资格均来自正式API与持久化事实。
- 中途刷新、一次失败重试和人工返修不会产生重复正式成果。
- PPTX能够在PowerPoint或WPS打开，关键文字、公式和标注保持可编辑。
- 脱敏记录基本成本、耗时、Provider请求和文件哈希，不要求首版完成全量运营观测。

## 7. 阶段5：自动化与产品化

MVP通过后增加：

- 完全自动审批策略。
- 多模型路由、限流和故障切换。
- 预算、质量与风险自动暂停。
- 内容包、提示词和工作流的管理端导入发布。
- 批量项目与批量生成。
- 完整权限、审计和运营能力。
- 性能、并发、缓存和成本优化。
- 更多学科、教材和内容包。
- 完整音频权利声明、逐镜头lineage/stale和90秒视频全节点验收。

完全自动不是第二套工作流，只是同一运行时上的审批与推进策略。

## 8. 并行策略

### 受控阶段重叠

当一个阶段只被独立的前端或消费者联调门禁阻塞时，可以通过Decision Issue授权下一阶段中不依赖该门禁的后端基础任务提前启动。受控重叠必须同时满足：

- 前一阶段和被阻塞任务继续保持未完成状态，不得用后端测试替代浏览器或消费者验收。
- 下一阶段使用独立Milestone、父Issue和可独立验收的子Issue；每个子Issue仍使用短分支、仓库外worktree和Draft PR。
- 合同、核心数据和统一状态机先于业务分支；依赖未满足的任务保持`status:ready`。
- 所有阶段合计同时进行的后端主任务不超过三个；普通CI使用确定性Fake，Provider变更和阶段出口使用真实冒烟。
- 发现产品、合同或数据语义冲突时停止受影响任务，回到Decision Issue裁决，不同时支持相反口径。

Decision #18曾授权在前端阻塞时提前推进后端基座；[Decision #199](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/199)结束该默认扩张期。现在除收口已活动的#170和缩减#118外，不再以“前端未完成”为由启动后续底座任务。

### 可以并行

- 前端基于已发布OpenAPI、Schema和MSW实现完整界面。
- #170与PR #111可以在不竞争文件和合同lease时分别收口。
- 普通测试、MVP浏览器验收准备和独立只读审查可以跟随所属任务推进。
- 自动化测试和最低安全检查跟随所属业务任务同步开发；可观测性扩张必须由真实MVP缺口驱动。

### 不能并行绕过

- 数据所有权未确定前，不分别设计多套表结构。
- 状态机未统一前，不分别实现PPT和视频任务状态。
- 项目资产合同未稳定前，不分别实现创作中心回写。
- Prompt、Schema和内容包没有版本规则前，不把模板写死进业务代码。

同时进行的后端主工作流不超过三个。并行团队通过合同和子Issue协作，不通过长期前端、后端或个人分支协作。

## 9. 阶段验收与调整

每个阶段结束必须具备：

- 可运行软件。
- 可演示用户结果。
- 自动化测试证据。
- 可执行数据迁移。
- 现行文档、合同和实现一致。
- 已知风险、成本和阻塞清单。
- 更新后的`CURRENT_STATUS.md`和下一阶段Issue集合。

路线允许根据真实实现调整。调整通过Decision Issue进入，确认后直接修改本文件和受影响的现行合同，不追加版本历史。
