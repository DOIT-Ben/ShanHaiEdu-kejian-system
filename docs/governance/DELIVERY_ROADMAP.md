# 渐进式开发与交付路线

目标不是先完成所有基础设施，也不是同时铺开所有模块，而是用最小公共基座支撑一条条可演示的真实用户链路。每个阶段都必须产生可运行结果，最终MVP必须体现整个小学数学课件平台，而不是一个孤立功能。

## 1. 开发原则

- 先建立最小共享基座，再按纵向业务链路交付。
- 按步骤辅助创作运行时先于完全无人值守的`automatic`策略。
- 一键推进到下一个审核点先于无人值守全流程。
- 前端可以基于合同和MSW整体推进；后端按可验收链路逐步集成。
- 公共合同稳定后，PPT和视频可以并行。
- 普通CI使用确定性Fake；阶段验收使用真实模型和真实文件。
- 不以Mock、截图或静态原型宣称里程碑完成。

### 固定责任分工

- Codex负责除生产前端之外的平台：`apps/api`、`workers`、`workflow`、`infra`、数据库、对象存储、模型网关、共享合同、生成客户端命令和后端CI。
- 独立前端承接方只负责`apps/web`、Figma、Storybook、MSW、前端测试、前端部署和真实API联调。
- Codex不得为了演示在`apps/web`堆临时页面；外包方不得实现服务端业务、数据库、任务队列或模型调用。
- 双方只通过OpenAPI、JSON Schema、SSE、错误合同、Mock场景和关联Issue协作。
- 前端发现合同缺口时提交复现、影响和建议，由Codex负责的合同任务统一修改；前端不得建立私有DTO或改变业务语义。

## 2. 阶段0：协作与工程基线

### 目标

让团队可以在同一口径下安全开发、审核、交接和清理。

### 范围

- README、AGENTS、CURRENT_STATUS和治理文档。
- Issue/PR模板、CODEOWNERS、标签和Milestone规则。
- `main`分支保护和合并后自动删除分支。
- Monorepo目标目录和基础开发命令。
- 前后端基础CI、合同校验和仓库清洁检查。

### 可见成果

新成员可以从Issue领取任务，启动开发环境，提交一个满足检查的PR，并由另一位成员按标准接手。

### 出口

- 治理规则已合并并经过一次真实任务验证。
- 前后端骨架、本地依赖和CI能够启动。
- OpenAPI、JSON Schema、格式、类型和基础测试进入自动门禁。

## 3. 阶段1：最小可运行基座

### 目标

打通浏览器、API、数据库、任务队列、对象存储和模型网关。

### 范围

- React应用接入真实API和MSW模式。
- FastAPI模块化单体、PostgreSQL、Redis、对象存储和Worker。
- 用户、项目、课时、资产、版本和任务基础模型。
- 文件上传、任务状态、SSE进度、取消和错误反馈。
- Prompt快照、模型调用审计和一个真实文本模型适配器。

### 可见成果

教师可以登录、创建项目、上传测试教材，并实时看到解析任务状态和项目资产记录。

### 出口

- 一条上传任务从前端到Worker再回到SSE完整运行。
- 失败、重试、取消和幂等至少有集成测试。
- 密钥只存在服务端受控环境。
- 数据库可通过Alembic从空库升级。

## 4. 阶段2：教材到教案纵向链路

### 目标

交付第一个真实教师价值闭环。

### 链路

```text
上传单知识点教材
→ PDF解析
→ 课时划分
→ 审核修改
→ 分课时教案生成
→ 动态结构编辑
→ 批准并版本化
→ 保存项目资产
```

### 要求

- 使用真实文本模型。
- 提示词可见、可改并保存快照。
- 默认12部分由内容定义驱动，不能写死在页面或数据库列中。
- 三类九套导入作为独立版本化附属成果。
- 每个节点支持返修、重新生成和保留历史版本。
- 内容包运行时先支持内置受控包；管理端导入界面可以后置。
- 首套内置包使用已批准的47节点目录（22个`model_generation`、13个`deterministic`、12个`human_gate`）及其输入、Prompt、输出、投影和质量/审核合同；黄金Fixture只作为确定性开发基线，阶段出口仍使用真实教材和真实文本模型。

### 启动依赖与实施波次

阶段2不能从通用执行器直接开工。先关闭Provider、当前合同、内容发布、公共Prompt、关系语义、Attempt恢复、Prompt收窄和#130输出投影门禁，再进入跨模块原子执行。每个Issue继续使用独立短分支和Draft PR；同一文件、同一机器合同或同一跨模块事务所有者存在竞争写入时，即使未达到三个任务上限也不得并发。

1. 首个动作只恢复[Issue #61](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/61)：从最新`main`校准既有Draft PR，确认Provider中立合同、确定性Fake和统一网关边界，不在旧准备提交上继续堆实现。
2. 第一波最多并行#61、[Issue #85](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/85)和[Issue #88](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/88)。三者分别拥有模型调用基础、当前/规划API分层和黄金ContentRelease发布，禁止跨入彼此所有权。
3. 第二波推进[Issue #51](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/51)、[Issue #92](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/92)为通用执行器拆出的最小应用接口/增长门禁子任务，以及[Decision #82](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/82)。#51必须在#85之后；#92不借机全量重构。
4. 第三波推进[Issue #86](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/86)、[Issue #90](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/90)和[Decision #83](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/83)。#86固定stale关系语义，#90固定Attempt恢复与并发序号，二者完成前不启动通用执行器。
5. [Issue #89](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/89)满足全部前置后先作为唯一后端主任务启动，使用真实PostgreSQL和确定性Fake关闭一个黄金文本节点的原子纵向链；Adapter真实冒烟和阶段真实文本出口单独执行，不能用Fake替代。
6. #89稳定后，[Issue #84](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/84)和[Issue #87](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/87)可以在无文件竞争时并行。破坏性列退役[Issue #91](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/91)必须等消费者归零并取得当次明确批准；治理兼容[Issue #94](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/94)只在PR #62进入终态后启动。

普通CI始终使用确定性Fake。#61的Provider Adapter变化、#89的阶段文本出口以及后续图片/视频阶段出口必须分别使用受控真实冒烟，并只记录脱敏Provider、模型、request ID、UTC、延迟、用量、成本和结论。

### 可见成果

教师从教材得到可编辑、可批准、可追溯的多课时教案和导入方案。

### 出口

- 一个黄金教材样例完成整条链路。
- 结构化JSON、教师可读投影和资产版本一致。
- 项目刷新或任务重试不会产生重复正式资产。

## 5. 阶段3：PPT与视频受控并行

阶段2确认项目、资产、任务、版本、Prompt和工作流合同后，拆成两个独立但共享基础能力的工作流。首套业务内容包合并后，PPT与视频可分别从批准教案和已选导入方案快照启动独立实现Issue；两条链无需等待彼此的媒体产物，但仍共同依赖项目运行时、模型网关、资产版本和原子保存。

### PPT工作流

```text
批准教案
→ PPT大纲
→ 逐页设计
→ 封面
→ 视觉规范
→ 正文图片
→ 页面预览与修改
→ 可编辑PPTX
```

验收重点：封面独立设计，正文纯白，每页有图像主视觉，关键文字、公式和标注保持可编辑。

### 视频工作流

```text
选定导入方案快照
→ 母版剧本
→ 粗分镜
→ 视觉母版
→ 四类图片资产
→ 细分镜与提示词
→ 候选视频片段
→ 保存正式片段
→ 音频字幕
→ MP4合成
```

验收重点：图片先于细分镜，候选结果被选择并保存后才产生正式`clip_id`，视频不得读取完整教案或教材。

### 共享能力

- 模型网关和Provider适配。
- Prompt编译和快照。
- 任务、预算、重试和SSE。
- 项目资产、版本、创作任务包和原子保存。
- 图片、视频与PPT通用创作中心。
- 审核、质量规则和真实产物验证。

PPT和视频不得复制共享能力或维护不同状态机。

## 6. 阶段4：端到端小学数学MVP

正式MVP必须完成一次真实项目：

```text
上传教材
→ 自动划分课时
→ 生成并批准教案
→ 选择三类九套中的一个导入
→ 生成完整PPT
→ 生成课堂导入视频
→ 保存图片与视频资产
→ 导出教案、PPTX和MP4
```

### MVP边界

- 只闭环小学数学和一个小知识点项目。
- 每类模型先使用一个稳定Provider。
- 先交付一套验证过的内容包。
- 先交付按步骤辅助创作和“一键推进到审核点”，两者引用同一工作流定义。
- 用户可以编辑业务提示词、节点内容和资产选择。
- PPTX真实可编辑，MP4由真实片段、音频和字幕合成。
- Mock只用于开发测试，不是验收结果。

### MVP出口

- 新教师无需离开平台完成黄金项目。
- 中途刷新、失败重试和人工返修不会破坏已批准成果。
- 交付物能从项目资产追溯到节点、Prompt、模型调用和输入快照。
- 成本、耗时、成功率和人工返修点可观测。

## 7. 阶段5：自动化与产品化

MVP通过后增加：

- 完全自动审批策略。
- 多模型路由、限流和故障切换。
- 预算、质量与风险自动暂停。
- 内容包、提示词和工作流的管理端导入发布。
- 批量项目与批量生成。
- 完整权限、审计和运营能力。
- 性能、并发、缓存和成本优化。
- 更多学科、教材和内容包。

完全自动不是第二套工作流，只是同一运行时上的审批与推进策略。

## 8. 并行策略

### 受控阶段重叠

当一个阶段只被独立的前端或消费者联调门禁阻塞时，可以通过Decision Issue授权下一阶段中不依赖该门禁的后端基础任务提前启动。受控重叠必须同时满足：

- 前一阶段和被阻塞任务继续保持未完成状态，不得用后端测试替代浏览器或消费者验收。
- 下一阶段使用独立Milestone、父Issue和可独立验收的子Issue；每个子Issue仍使用短分支、仓库外worktree和Draft PR。
- 合同、核心数据和统一状态机先于业务分支；依赖未满足的任务保持`status:ready`。
- 所有阶段合计同时进行的后端主任务不超过三个；普通CI使用确定性Fake，Provider变更和阶段出口使用真实冒烟。
- 发现产品、合同或数据语义冲突时停止受影响任务，回到Decision Issue裁决，不同时支持相反口径。

当前应用由[Decision #18](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/18)授权：阶段0的Issue #2/#11保持未完成，阶段1后端按[父Issue #19](https://github.com/DOIT-Ben/ShanHaiEdu-kejian-system/issues/19)的依赖波次推进，不修改或代写`apps/web`。

### 可以并行

- 前端基于已发布OpenAPI、Schema和MSW实现完整界面。
- 阶段1完成后，内容运行时与媒体Provider适配可以并行。
- 阶段2合同稳定后，PPT与视频工作流可以并行。
- 自动化测试、可观测性和安全检查跟随所属业务任务同步开发。

### 不能并行绕过

- 数据所有权未确定前，不分别设计多套表结构。
- 状态机未统一前，不分别实现PPT和视频任务状态。
- 项目资产合同未稳定前，不分别实现创作中心回写。
- Prompt、Schema和内容包没有版本规则前，不把模板写死进业务代码。

同时进行的后端主工作流不超过三个。并行团队通过合同和子Issue协作，不通过长期前端、后端或个人分支协作。

## 9. 阶段验收与调整

每个阶段结束必须具备：

- 可运行软件。
- 可演示用户结果。
- 自动化测试证据。
- 可执行数据迁移。
- 现行文档、合同和实现一致。
- 已知风险、成本和阻塞清单。
- 更新后的`CURRENT_STATUS.md`和下一阶段Issue集合。

路线允许根据真实实现调整。调整通过Decision Issue进入，确认后直接修改本文件和受影响的现行合同，不追加版本历史。
