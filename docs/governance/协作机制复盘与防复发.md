# 协作机制复盘与防复发

> 负责人：仓库工程负责人
>
> 读者：主智能体、独立审查者和维护者
>
> 唯一入口：`docs/governance/协作机制复盘与防复发.md`
>
> 更新条件：同类失败再次出现、现行门禁变化或残余风险被验证
>
> 替换规则：原位修改；机制失效时在同一PR删除，不保留版本、日期或附录副本

本文只保存仍会影响未来任务的协作机制，不保存完整会话、日常进度或一次性操作日志。具体任务状态仍以GitHub Issue和Pull Request为准。

## 1. 权威结论

本阶段最深根因不是多智能体、Windows或任务规模，而是把可从Git、GitHub和文件系统直接取得的状态，重新手工抄写成文本或交给带隐式副作用的组合命令处理。结果是代码可以正确，审查SHA、PR正文、规模判断和worktree状态却仍可能失真。

反事实成立：如果SHA、diff规模和远端声明由机器读取并校验，合并、main复验和物理清理由独立状态转换完成，已发生的元数据错误会在提交前被拒绝，worktree残留也会在清理前被预防或更早暴露。

合理推断：上述错误共同受“手工重写机器事实”和“把多个状态折叠进一个命令”驱动。竞争解释“任务太多”“Windows不稳定”“子智能体协作复杂”只能解释触发频率，不能解释为什么错误没有在提交或合并前被拒绝。

未验证项：新的worktree清理顺序尚未在下一个含PNPM依赖的任务收尾中实跑；在获得一次新鲜成功证据前只能标记为已落实待验证。

## 3. 根因链

### 3.1 审查证据漂移

```text
短SHA或当前印象
-> 主智能体手工抄写完整SHA
-> PR正文没有校验实际base/head
-> 审查报告可能绑定错误对象
-> 代码通过但批准证据不成立
```

失效门禁是PR正文只要求填写字段，却没有验证字段值。替代动作是从GitHub读取OID，并让CI校验批准态SHA与当前PR完全一致。

### 3.2 远端正文污染

```text
多行Markdown进入PowerShell插值字符串
-> 反引号、美元符号或换行被Shell解释
-> gh收到的正文与本地预期不同
-> 交接证据仍存在但不可可靠阅读或复用
```

失效门禁是只相信命令退出码，没有回读远端正文。替代动作是使用不插值的正文载体，并把回读加到同一逻辑切片验证中。

### 3.3 合并与清理副作用

```text
组合命令同时合并和删除分支
-> CLI隐式改变当前worktree
-> 依赖目录又阻断物理删除
-> Git登记、分支和磁盘目录进入不同状态
-> “已清理”声明早于事实
```

失效门禁是把PR、ref、worktree登记和物理目录当成一个状态。替代动作是逐项验证，先main复验，再单独清理本地资源。

### 3.4 规模判断偏差

```text
只扫描单文件行数
-> 得到“接近400行”的局部结论
-> 未计算PR文件数和净新增行数
-> 缺少评审导航
-> 审查成本和遗漏面被低估
```

失效门禁是规则存在但无diff级机器声明。替代动作是用原始diff保守触发规模门禁，再由评审导航解释生成物和迁移例外。

## 4. 主智能体可控责任

- 我本可直接读取完整OID，却手工补全短SHA；这是证据生成方式错误，不是GitHub问题。
- 我本可通过正文文件或结构化API提交Markdown，却使用了容易被PowerShell解释的字符串；命令成功不等于远端正文正确。
- 我本可把合并、远端删除、本地切换和物理清理拆开验证，却使用了带本地副作用的组合命令。
- 我知道worktree包含PNPM依赖，却没有在`git worktree remove`前清除可再生长路径目录。
- 我检查了单文件规模，却没有同步计算PR总规模，导致汇报口径不完整。

## 6. 现行执行顺序

### 6.1 更新PR证据

1. 用GitHub读取`baseRefOid`和`headRefOid`，不用短SHA推算。
2. 通过UTF-8正文文件、标准输入或结构化API提交多行Markdown。
3. 重新读取PR正文，确认复选框、SHA、换行和反引号未漂移。
4. 对远端正文运行`check_pull_request_status.py`。
5. Head变化时把批准恢复为pending，由同一审查者复核并重新绑定最终Head。

### 6.2 单轮审查

1. 主智能体在实现完成后指定一个未参与实现的只读审查者。
2. 审查者对精确base-to-head diff给出findings-first报告。
3. 主智能体修复findings并推送；这仍属于同一engagement。
4. 同一审查者复核修复、检查新增delta并确认最终完整diff。
5. 原审查者不可用或范围实质变化时才更换，并在PR记录原因。

### 6.3 合并与worktree清理

1. required checks、最终审查声明和PR正文回读全部通过。
2. 执行Squash Merge，不让命令隐式删除或切换本地分支。
3. 重新读取PR `MERGED`、merge commit、远端分支和当前worktree状态。
4. 在专用worktree切到detached `origin/main`并执行main复验。
5. 核验绝对路径和干净状态，先清可再生依赖与缓存。
6. 使用Git长路径模式移除worktree，prune后再删本地任务分支。
7. 最后分别证明PR已合并、Issue已关闭、远端分支消失、本地分支消失、worktree登记和物理目录都消失。

## 7. 残余风险

- PR规模机器门禁使用全部文件的原始统计，可能对生成物较多的PR保守触发；这是有意设计，评审导航负责解释，不应通过放宽脚本隐藏规模。
- 审查者身份独立性无法由本地脚本证明；项目以主智能体分配记录、只读约束和findings证据保证可追溯性。
- Windows重解析点和ACL组合仍可能让物理清理失败；失败时不得扩大删除范围，必须保留“Git已注销但目录尚在”的准确状态。
- 历史PR正文和评论不是现行规则源；本文件只提炼机制，不能用历史案例覆盖当前`AGENTS.md`和`TEAM_WORKFLOW.md`。
