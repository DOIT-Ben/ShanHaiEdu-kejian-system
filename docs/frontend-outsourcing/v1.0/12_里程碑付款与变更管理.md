# 12 里程碑、付款与变更管理

## 1. 建议周期

建议12–16周，实际工期由承接方在不削减范围的前提下报价。

## 2. 建议里程碑和付款

| 里程碑 | 交付 | 建议付款比例 |
|---|---|---:|
| M0 启动与技术验证 | 项目骨架、CI、Mock运行、组件策略和关键风险清单 | 5% |
| M1 Figma与设计系统 | 完整信息架构、核心高保真、组件库和交互原型 | 15% |
| M2 前端平台底座 | 登录、路由、布局、API客户端、错误、SSE、上传和Storybook | 15% |
| M3 文本生产闭环 | 项目、教材、课时、教案、三类九套、提示词和版本 | 25% |
| M4 PPT与视频工作区 | PPT页面、资产、分镜、镜头、片段、字幕和成片审核 | 20% |
| M5 管理端 | 模板中心、模型网关、预算、工作流和审计 | 10% |
| M6 联调验收与交接 | 真实API、测试、性能、安全、部署、源码和培训 | 10% |

付款以验收通过为前提，不以“页面基本完成”或“已经演示”替代验收。

## 3. 每个里程碑验收材料

- 本里程碑范围清单。
- Figma和Storybook链接。
- PR或发布版本。
- 自动化测试报告。
- 目标浏览器截图或录屏。
- 已知问题清单。
- 与本规范的差异说明。
- 下一阶段风险。

## 4. 缺陷等级

- P0：安全泄露、数据破坏、无法登录或核心系统不可用。
- P1：核心流程无法完成、付费重复、审批错误或错误资产引用。
- P2：重要功能异常但有明确绕行方案。
- P3：轻微视觉、文案和低频体验问题。

里程碑验收要求P0和P1为零。

## 5. 变更流程

任何范围变化必须提交变更单，包含：

- 变更原因。
- 当前行为和目标行为。
- 影响页面、组件、API、测试和文档。
- 工期和费用变化。
- 是否影响已验收功能。
- 回滚方案。
- 甲乙双方批准记录。

口头、聊天截图和会议中临时提出的想法不自动构成范围变更。

## 6. 范围澄清

以下不属于新增需求，应包含在原报价：

- 已规定的空、加载、错误、权限和响应式状态。
- 为通过自动化测试、类型检查和构建所需的修复。
- 与批准Figma不一致的修复。
- OpenAPI约定内的接口适配。
- 浏览器兼容、键盘可用和基础可访问性。
- 因承接方技术选型造成的重构。

以下通常需要变更单：

- 新角色或全新业务模块。
- 新增本规范未包含的编辑器类型。
- 修改工作流核心顺序。
- 增加移动端完整创作能力。
- 增加全局暗色模式。
- 后端合同发生破坏性变化且无法版本化处理。

## 7. 项目管理

- 甲方仓库从第一天开始保存代码，不接受项目结束时一次性交源码。
- 每周至少一次可运行版本和书面周报。
- 关键交互和组件先在Storybook验收，再进入大页面。
- 所有PR至少一名承接方资深工程师自审。
- 甲方可在任何里程碑进行代码和设计审查。

## 8. 退出与移交

合同终止或更换承接方时，当前已付款里程碑的源码、设计、文档、测试、账号清单和未完成事项必须完整移交。不得通过私有包、私有字体、外包方服务器或不可转移许可证形成锁定。
